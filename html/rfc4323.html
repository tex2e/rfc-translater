2. Overview
2. 概要

This MIB module provides a set of objects required for the management of DOCSIS 1.1 and 2.0 compliant Cable Modems (CM) and Cable Modem Termination Systems (CMTS). The specification is derived from the DOCSIS 2.0 Radio Frequency Interface specification [4]. Please note that the referenced DOCSIS specifications only requires Cable Modems to process IPv4 customer traffic. Design choices in this MIB module reflect those requirements. Future versions of the DOCSIS standard are expected to require support for IPv6 as well.

このMIBモジュールは、DOCSIS 1.1および2.0に準拠したケーブルモデム（CM）およびケーブルモデム終了システム（CMT）の管理に必要な一連のオブジェクトを提供します。仕様は、DOCSIS 2.0無線周波数インターフェイス仕様[4]から導出されます。参照されているDOCSIS仕様には、IPv4顧客トラフィックを処理するためのケーブルモデムのみが必要であることに注意してください。このMIBモジュールの設計の選択は、これらの要件を反映しています。DOCSIS標準の将来のバージョンは、IPv6のサポートも必要とされると予想されます。

The key words "MUST", "MUST NOT", "REQUIRED", "SHALL", "SHALL NOT", "SHOULD", "SHOULD NOT", "RECOMMENDED", "MAY", and "OPTIONAL" in this document are to be interpreted as described in RFC 2119 [5].

「必須」、「そうしない」、「必須」、「shall」、「shall "、" ingle "、" should "、" not "、" becommended "、" bay "、および「optional」は、RFC 2119 [5]に記載されているように解釈される。

2.1. Textual Conventions
2.1. テキストの慣習

The textual convention "DocsIetfQosRfMacIfDirection" is defined to indicate the direction of a packet classifier relative to an interface. It takes the values of either downstream(1) or upstream(2).

テキスト条約「docsietfqosrfmacifdirection」は、インターフェイスに対するパケット分類器の方向を示すように定義されています。下流（1）または上流（2）のいずれかの値を取得します。

2.2.1. docsIetfQosPktClassTable
2.2.1. docsietfqospktclasStable

The docsIetfQosPktClassTable reports the Service Flow Classifiers implemented by the managed device. The table is indexed by the tuple { ifIndex, docsIetfQosServiceFlowId, docsIetfQosPktClassId }. The ifIndex corresponds to a CATV MAC interface. Each CATV MAC interface has a set of Service Flows identified with a docsIetfQosServiceFlowId value that is unique for that interface. Each Service Flow may have a number of packet classifiers that map packets to the flow. The ClassifierId for the classifier is unique only within a particular Service Flow.

docsietfqospktclasstableは、管理されたデバイスによって実装されたサービスフロー分類器を報告しています。このテーブルは、tuple {ifindex、docsietfqosserviceflowid、docsietfqospktclassid}によってインデックス付けされています。IFINDEXは、CATV MACインターフェイスに対応しています。各CATV MACインターフェイスには、そのインターフェイスに固有のdocsietfqosserviceflowid値で識別される一連のサービスフローがあります。各サービスフローには、パケットをフローにマッピングする多数のパケット分類器があります。分類器の分類子は、特定のサービスフロー内でのみ一意です。

The semantics of packet classification are provided in [4]. Briefly, the DOCSIS MAC interface calls for matching packets based on values within the 802.2 (LLC), 802.3, IP, and/or UDP/TCP headers. Packets that map more than one classifier are prioritized according to their docsIetfQosPktClassPriority values. The docsIetfQosServiceFlowId (an index object) indicates to which Service Flow the packet is classified.

2.2.8. docsietfqosserviceclasstable

This table defines the Service Class Name and references a QoS Parameter Set for each Service Class defined in a CMTS. It is indexed by the Service Class Name string itself. The table is read-create on a CMTS, and is not implemented in a CM. Each entry of the docsIetfQosServiceClassTable should define a template for flows in a given direction (upstream or downstream). Some parameters of the docsIetfQosServiceClassTable are specific to a particular direction, and so their values are not applicable when used as a template for flows in the other direction.

2.2.9. docsIetfQosServiceClassPolicyTable
2.2.9. docsietfqosserviceclasspolicytable

The docsIetfQosServiceClassPolicyTable can be referenced by the docsDevFilterPolicyTable of [6] in order to have a "policy" that classifies packets to a named Service Class. 2.2.10. docsIetfQosPHSTable
2.2.10. docsietfqosphstable

The Payload Header Suppression (PHS) feature of DOCSIS 1.1 and 2.0 permits packets to replace the unchanging bytes of the Ethernet, IP, and UDP headers with a one-byte index when transmitting on the cable network. This is especially useful for IP Telephony packets, where such suppression can result in almost twice the number of calls supported within the same upstream channel.

DOCSIS 1.1および2.0のペイロードヘッダー抑制（PHS）機能により、パケットは、ケーブルネットワーク上で送信するときにイーサネット、IP、およびUDPヘッダーの変化しないバイトを1バイトインデックスに置き換えることができます。これは、同じ上流チャネル内でサポートされているコールの数のほぼ2倍をもたらす可能性のあるIPテレフォニーパケットに特に役立ちます。

Each entry of the table corresponds to a PHS Rule as described in section 8.4 of [4]. The rules are identified by their corresponding Service Flow ID and docsIetfQosPktClassId. A PHS rule is associated with exactly one classifier. The table is therefore indexed by the tuple { ifIndex, docsIetfQosServiceFlowId, docsIetfQosPktClassId}.

表の各エントリは、[4]のセクション8.4で説明されているPHSルールに対応しています。ルールは、対応するサービスフローIDおよびdocsietfqospktclassidによって識別されます。PHSルールは、正確に1つの分類子に関連付けられています。したがって、テーブルはtuple {ifindex、docsietfqosserviceflowid、docsietfqospktclassid}によってインデックス付けされています。

This table is read-only, and MUST be implemented on both the CM and CMTS when PHS is supported.

このテーブルは読み取り専用であり、PHSがサポートされている場合はCMとCMTの両方に実装する必要があります。

2.2.11 docsIetfQosCmtsMacToSrvFlowTable
2.2.11 docsietfqoscmtsmactosrvflowtable

ダウンストリームマルチキャストサービスフローは、1つのcmだけに関連付けられていないため、docsietfqoscmtsmactosrvflowtableには示されていません。

3. Externally Administered Classification
3. 外部で管理された分類

DOCSIS 1.1 and 2.0 provide rich semantics for the classification of packets to Service Flows with the Service Flow Classifier table. Service Flow Classifiers may be created statically in the DOCSIS CM configuration file, or may be created dynamically with Dynamic Service Addition (DSA) and Dynamic Service Change (DSC) DOCSIS MAC messages.

DOCSIS 1.1および2.0は、サービスフロー分類器テーブルを使用して、パケットの分類に豊富なセマンティクスを提供します。サービスフロー分類器は、DocSIS CM構成ファイルで静的に作成される場合や、動的サービス追加（DSA）および動的サービス変更（DSC）DocSIS Macメッセージを使用して動的に作成される場合があります。

Several major issues arose with the concept of externally administered classification; e.g., should an external SNMP manager be permitted to create classification rows? One problem was the coordination of classifier IDs because such an approach would require either separate classifier ID number spaces or objects to coordinate both internal and external classifier ID assignments. A more serious problem, however, was that external creation of SF Classifiers would require "knowledge" of the individual Service Flow ID for Service Flows by external applications. It was strongly felt by the committee that SFIDs should remain internal DOCSIS objects, and not be transmitted as part of protocol flows, e.g., for IP packet telephony signaling. DOCSIS 1.1 introduced the concept of named Service Classes for ease of administration within a domain of CMs and CMTSs. What was desired was to permit external classification of packets to a Service Class, not to a particular Service Flow.

外部で管理された分類の概念には、いくつかの主要な問題が発生しました。たとえば、外部SNMPマネージャーは分類行を作成することを許可されるべきですか？1つの問題は、このようなアプローチでは、内部および外部分類器ID割り当ての両方を調整するために、個別の分類器ID番号スペースまたはオブジェクトのいずれかを必要とするため、分類器IDの調整でした。しかし、より深刻な問題は、SF分類子の外部作成が、外部アプリケーションによるサービスフローの個々のサービスフローIDの「知識」を必要とすることです。委員会は、SFIDSは内部DOCSISオブジェクトのままであり、IPパケットテレフォニーシグナリングのためにプロトコルフローの一部として送信されないことを強く感じました。DOCSIS 1.1は、CMSおよびCMTSSのドメイン内での管理容易さのために、名前付きサービスクラスの概念を導入しました。望まれたのは、特定のサービスフローではなく、サービスクラスにパケットの外部分類を許可することでした。

The DOCSIS committee therefore decided to use the already-defined IP Packet Filter Table [6] for the external classification of packets for QoS purposes. The docsDevIpPacketFilterTable defines similar packet matching criteria as does docsIetfQosPktClassTable, but it matches a packet to an arbitrary "policy set" instead of a particular Service Flow. One of the policies in the policy set then selects the Service Class of the SF on which to forward the packet. The docsIetfQosServiceClassPolicyTable of this MIB module defines the Service Class Name to which a packet is classified.

したがって、DOCSIS委員会は、QoS目的でパケットの外部分類に、すでに定義されたIPパケットフィルターテーブル[6]を使用することを決定しました。docsdevippacketfiltertableは、docsietfqospktclasstableと同様のパケットマッチング基準を定義しますが、特定のサービスフローの代わりにパケットを任意の「ポリシーセット」と一致させます。ポリシーセットのポリシーの1つは、パケットを転送するためにSFのサービスクラスを選択します。このMIBモジュールのdocsietfqosserviceclasspolicytableは、パケットが分類されるサービスクラス名を定義します。

4. DOCSIS and IPv4 Type-of-Service (ToS) Field
4. DOCSISおよびIPv4タイプオブサービス（TOS）フィールド

The DOCSIS-IETF-QOS-MIB MIB module relies on the DOCSIS MAC layer protocols and uses objects that reflect the IPv4 Type-of-Service (ToS) octet as defined in [14]. The applicability of these objects is limited to the DOCSIS access network. The past and current versions of the DOCSIS specifications for which this MIB module is defined do not reflect Differentiated Services [9] on the DOCSIS access network. However, with proper selection of values for these objects, the network operator can enforce Differentiated Services Per-hop Behaviors (PHBs) on the DOCSIS Access Network, and can configure the modification of the DSCP for certain packet flows as they enter the metro network from the access network. Essentially this makes the DOCSIS access network TOS marking compatible with the wider use of DSCP outside DOCSIS networks. Note that because the entire IPv4 TOS octet may be available for modification via the latter mechanism (due to the current MAC level DOCSIS protocols and CLI interface configuration), it is possible that the DOCSIS network could be configured to modify the Explicit Congestion Notification (ECN) bits [10] of certain packets. This modification of the ECN bits is prevented by the MIB module's design. The MIB module prohibits the modification of the TOS octet (read-only objects: docsIetfQosPktClassIpTosLow, docsIetfQosPktClassIpTosHigh docsIetfQosPktClassIpTosMask, docsIetfQosParamSetTosAndMask, docsIetfQosParamSetTosOrMask) and allows the DSCP field to be modified (read-create object: docsIetfQosServiceClassDSCPOverwrite).

docsis-etf-qos-mib mibモジュールは、[14]で定義されているように、docsis macレイヤープロトコルに依存し、IPv4タイプオブサービス（TOS）オクテットを反映するオブジェクトを使用します。これらのオブジェクトの適用性は、DocSISアクセスネットワークに限定されています。このMIBモジュールが定義されているDOCSIS仕様の過去および現在のバージョンは、DOCSISアクセスネットワーク上の差別化されたサービス[9]を反映していません。ただし、これらのオブジェクトの値を適切に選択することで、ネットワークオペレーターは、DOCSISアクセスネットワークに差別化されたサービス（PHB）を実施することができ、メトロネットワークに入るときに特定のパケットフローのDSCPの変更を構成できます。アクセスネットワーク。基本的に、これにより、DOCSISアクセスネットワークTOSは、DOCSISネットワーク以外でのDSCPをより広く使用することと互換性があります。IPv4 TOSオクテット全体が後者のメカニズム（現在のMACレベルのDOCSISプロトコルとCLIインターフェイス構成により）を介して変更できるため、DOCSISネットワークを明示的な混雑通知を変更するように構成できる可能性があることに注意してください（ECN）特定のパケットのビット[10]。このECNビットの変更は、MIBモジュールの設計によって防止されます。MIBモジュールは、TOSオクテットの変更を禁止しています（読み取り専用オブジェクト：docsietfqospktclassiptoslow、docsietfqospktclassiptoshigh docsietfqospktclassiptossak、docsietfqosparamsettosandmask、docsietfqosparametsosormaskのdcsparametsetosormask） erviceclassdscpoverwrite）。

6. Security Considerations
6. セキュリティに関する考慮事項

This MIB module relates to an agent that will provide metropolitan public Internet access. As such, improper manipulation of the objects represented by this MIB module may result in denial of service to a large number of end-users [6]. Manipulation of the docsIetfQosServiceClassTable and docsIetfQosServiceClassPolicyTable may allow an end-user to increase his or her service levels, or affect other end-users in either a positive or negative manner. In addition, manipulation of docsIetfQosServiceFlowLogControl could allow an attacker to remove logs of packet and byte counts forwarded on a Service Flow. If such logs were used for billing, the attacker would obtain free service.

このMIBモジュールは、大都市のパブリックインターネットアクセスを提供するエージェントに関連しています。そのため、このMIBモジュールで表されるオブジェクトの不適切な操作により、多数のエンドユーザーに対するサービスの拒否が生じる可能性があります[6]。docsietfqosserviceclasstableおよびdocsietfqosserviceclasspolicytableの操作により、エンドユーザーはサービスレベルを上げたり、他のエンドユーザーにポジティブまたはネガティブな方法で影響を与えることができます。さらに、docsietfqosserviceflowlogcontrolの操作により、攻撃者はサービスフローに転送されたバイト数とバイト数のログを削除できるようになります。そのようなログが請求に使用された場合、攻撃者は無料のサービスを取得します。

There are a number of management objects defined in this MIB module with a MAX-ACCESS clause of read-write and/or read-create. Such objects may be considered sensitive or vulnerable in some network environments. The support for SET operations in a non-secure environment without proper protection can have a negative effect on network operations. These are the tables and objects and their sensitivity/vulnerability:

このMIBモジュールには、読み取りワイトおよび/またはread-Createの最大アクセス句を備えた管理オブジェクトが多数あります。このようなオブジェクトは、一部のネットワーク環境で敏感または脆弱と見なされる場合があります。適切な保護なしの非セキュア環境でのセット操作のサポートは、ネットワーク操作に悪影響を与える可能性があります。これらはテーブルとオブジェクトであり、その感度/脆弱性です。

o The docsIetfQosServiceClassTable provides a template of QOS parameters such as maximum rate limits for a named service class. Changing these parameters would allow an attacker to obtain an unauthorized class of service.

o docsietfqosserviceclasStableは、指定されたサービスクラスの最大レート制限など、QoSパラメーターのテンプレートを提供します。これらのパラメーターを変更すると、攻撃者が許可されていないサービスのクラスを取得できます。

o The docsIetfQosServiceClassPolicyTable applies CMTS vendor proprietary policies for packet forwarding, including dropping, scheduling, notification, or other policies. Changing this table could allow an attacker to deny service to all subscribers of the CMTS or could grant the attacker unauthorized forwarding policies.

o docsietfqosserviceclasspolicytableは、ドロップ、スケジューリング、通知、またはその他のポリシーなど、パケット転送のためのベンダー専有ポリシーを適用します。このテーブルを変更すると、攻撃者がCMTのすべてのサブスクライバーへのサービスを拒否したり、攻撃者に不正な転送ポリシーを付与することができます。

o The docsIetfQosServiceFlowLogControl object controls the deletion of entries in the docsIetfQosServiceFlowLogTable, which acts as a historical "detail record" of DOCSIS Service Flow packets and bytes transmitted. Such records may be used for billing purposes, so the unauthorized deletion of the records can result in free service.

o docsietfqosserviceflowlogcontrolオブジェクトは、docsietfqosserviceflowlogtableのエントリの削除を制御します。このようなレコードは請求目的で使用される場合があるため、レコードの不正な削除により無料のサービスが発生する可能性があります。

Some of the readable objects in this MIB module (i.e., objects with a MAX-ACCESS other than not-accessible) may be considered sensitive or vulnerable in some network environments. It is thus important to control even GET access to these objects and possibly to even encrypt the values of these objects when sending them over the network via SNMP. These are the tables and objects and their sensitivity/vulnerability:

このMIBモジュールの読み取り可能なオブジェクトのいくつか（つまり、アクセスできないこと以外に最大アクセスを備えたオブジェクト）は、一部のネットワーク環境で敏感または脆弱と見なされる場合があります。したがって、これらのオブジェクトにアクセスしても、SNMPを介してネットワーク上に送信する際にこれらのオブジェクトの値を暗号化することさえ制御することが重要です。これらはテーブルとオブジェクトであり、その感度/脆弱性です。

o Unauthorized SNMP GET access of the docsIetfQosPktClassTable or docsIetfQosPHSTable can allow an attacker to learn IP addresses permitted to have enhanced quality of service, for possible spoofing. This table typically contains the IP addresses involved in voice-over-IP sessions, for example.

o 許可されていないSNMPは、docsietfqospktclasStableまたはdocsietfqosphstableにアクセスすることで、攻撃者がスポーフの可能性のあるサービス品質を強化することが許可されたIPアドレスを学習できるようにします。このテーブルには、通常、音声オーバーIPセッションに関連するIPアドレスが含まれています。

o Unauthorized SNMP GET access of the docsIetfQosParamSetTable allows an attacker to learn the names of Service Classes that are permitted to have enhanced QoS service, and the values of that enhanced service. That name can be referenced in an unauthorized DOCSIS cable modem configuration file to obtain enhanced service.

o 許可されていないSNMPは、docsietfqosparamsettableにアクセスすることで、攻撃者がQoSサービスを強化することが許可されているサービスクラスの名前とその強化されたサービスの値を学習することができます。その名前は、強化されたサービスを取得するために、許可されていないDocsisケーブルモデム構成ファイルで参照できます。

o Unauthorized SNMP GET access of the docsIetfQosServiceFlowTable can tell an attacker when Service Flows are active, e.g., when a voice-over-IP call is in progress.

o 許可されていないSNMPは、DocsietfQosserviceFlowtableにアクセスできます。たとえば、サービスフローがアクティブである場合、たとえば、音声オーバーIPコールが進行中の場合に攻撃者に伝えることができます。

Unauthorized SNMP GET access of the docsIetfQosServiceFlowLogTable can expose private information about network usage.

許可されていないSNMPは、docsietfqosserviceflowlogtableにアクセスすると、ネットワークの使用に関する個人情報を公開できます。

o Unauthorized SNMP GET access of the docsIetfQosServiceFlowStatsTable, docsIetfQosUpstreamStatsTable, docsIetfQosDynamicServiceStatsTable, docsIetfQosServiceFlowLogTable, and docsIetfQosCmtsMacToSrvFlowTable can tell an attacker the volume of traffic to and from any Service Flow in the system, resulting in loss of privacy of the amount and direction of data transfer.

o 不正なSNMPは、docsietfqosserviceflowstatstableにアクセスし、docsietfqosupstreamstatstable、docsietfqosdynamicestattable、docsietfqosserviceflowlogtable、およびdocsietfqoscmtsmactosrvflowtableが潜在系を攻撃することができるようになります。データ転送の。

SNMP versions prior to SNMPv3 did not include adequate security. Even if the network itself is secure (for example by using IPSec), even then, there is no control as to who on the secure network is allowed to access and GET/SET (read/change/create/delete) the objects in this MIB module. It is RECOMMENDED that implementers consider the security features as provided by the SNMPv3 framework (see [15], section 8), including full support for the SNMPv3 cryptographic mechanisms (for authentication and privacy). Further, deployment of SNMP versions prior to SNMPv3 is NOT RECOMMENDED. Instead, it is RECOMMENDED to deploy SNMPv3 and to enable cryptographic security. It is then a customer/operator responsibility to ensure that the SNMP entity giving access to an instance of this MIB module, is properly configured to give access to the objects only to those principals (users) that have legitimate rights to indeed GET or SET (change/create/delete) them.

SNMPV3以前のSNMPバージョンには、適切なセキュリティは含まれていませんでした。ネットワーク自体が（たとえばIPSECを使用して）安全である場合でも、それでもセキュアネットワークで誰がアクセスして取得/セット（読み取り/変更/作成/削除/削除）を制御することはできません。MIBモジュール。SNMPV3暗号化メカニズム（認証とプライバシーのため）の完全なサポートを含む、実装者はSNMPV3フレームワークで提供されるセキュリティ機能を考慮することをお勧めします（[15]、セクション8を参照）。さらに、SNMPV3より前のSNMPバージョンの展開は推奨されません。代わりに、SNMPV3を展開し、暗号化セキュリティを有効にすることをお勧めします。その場合、このMIBモジュールのインスタンスへのアクセスを提供するSNMPエンティティが、実際に取得または設定する正当な権利を持つプリンシパル（ユーザー）にのみオブジェクトにアクセスできるように適切に構成されていることを保証するのは、顧客/オペレーターの責任です（それらを変更/作成/削除）それら。
