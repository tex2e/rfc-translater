1. Introduction
1. はじめに

The Network Time Protocol (NTP) and Network Time Security (NTS) documents define a number of registries, collectively called the NTP registries. The NTP registries can all be found at <https://www.iana.org/assignments/ntp-parameters> and the NTS registries can all be found at <https://www.iana.org/assignments/ nts>.

ネットワークタイムプロトコル（NTP）およびネットワークタイムセキュリティ（NTS）ドキュメントは、NTPレジストリと集合的に呼ばれる多くのレジストリを定義しています。NTPレジストリはすべて<https://www.iana.org/assignments/ntp-parameters>にあり、NTSレジストリはすべて<https://www.iana.org/assignments/ nts>にあります。

Some registries are correct, but some include incorrect assignments and some don't follow common practice. For the sake of completeness, this document reviews all NTP and NTS registries, and corrects the registries where necessary.

いくつかのレジストリは正しいですが、一部のレジストリには誤った割り当てが含まれており、一部は一般的な慣行に従わないものもあります。完全性のために、このドキュメントはすべてのNTPおよびNTSレジストリをレビューし、必要に応じてレジストリを修正します。

The bulk of this document can be divided into two parts:

このドキュメントの大部分は、2つの部分に分けることができます。

* a summary of the relevant registries, including syntax requirements, registration procedures, and the defining documents.

* 構文要件、登録手順、および定義文書など、関連するレジストリの概要。

* a revised format and entries for each registry being modified.

* 変更されている各レジストリの改訂された形式とエントリ。

2. Existing Registries
2. 既存のレジストリ

This section describes the registries and the rules for them. It is intended to be a short summary of the syntax and registration requirements for each registry. The semantics and protocol processing rules for each registry -- that is, how an implementation acts when sending or receiving any of the fields -- are not described here.

このセクションでは、レジストリとそれらのルールについて説明します。各レジストリの構文と登録要件の短い要約になることを目的としています。各レジストリのセマンティクスとプロトコル処理ルール、つまり、フィールドを送信または受信するときに実装がどのように機能するかは、ここでは説明されていません。

2.1. Reference ID and Kiss-o'-Death Registries
2.1. 参照IDおよびKISS-O'-DEATHレジストリ

[RFC5905] defines two registries: "NTP Reference Identifier Codes" in Section 7.3 and the "NTP Kiss-o'-Death Codes" in Section 7.4. Reference identifiers and kiss codes can be up to four ASCII characters, padded on the right with all-bits-zero if necessary. Entries that start with 0x58, the ASCII letter uppercase X, are reserved for Private or Experimental Use. Both registries are First Come First Served. The registries were created per Section 16 of [RFC5905].

[RFC5905]は、セクション7.3の「NTP参照識別子コード」とセクション7.4の「NTP KISS-O'-DEATHコード」を定義します。参照識別子とKISSコードは、必要に応じて右側に右側に右にパッド入り、最大4つのASCII文字になります。0x58で始まるASCIIレター大文字Xから始まるエントリは、プライベートまたは実験的使用のために予約されています。両方のレジストリが最初に提供されます。レジストリは、[RFC5905]のセクション16に従って作成されました。

2.2. Extension Field Types
2.2. 拡張フィールドタイプ

Section 7.5 of [RFC5905] defines the on-the-wire format of extension fields but does not create a registry for them.

[RFC5905]のセクション7.5は、拡張フィールドのオンザワイヤー形式を定義しますが、それらのレジストリを作成しません。

Section 13 of [RFC5906] mentions the "NTP Extension Field Types" registry, and defines it indirectly by defining 30 extensions (10 each for request, response, and error response). It does not provide a formal definition of the columns in the registry. Section 10 of [RFC5906] splits the Field Type into four subfields, only for use within the Autokey extensions.

[RFC5906]のセクション13は、「NTP拡張フィールドタイプ」レジストリに言及し、30の拡張機能（リクエスト、応答、およびエラー応答のためにそれぞれ10）を定義することにより間接的に定義します。レジストリ内の列の正式な定義は提供されません。[RFC5906]のセクション10は、Autokey拡張機能内でのみ使用するために、フィールドタイプを4つのサブフィールドに分割します。

[RFC7821] adds a new entry, Checksum Complement, to the "NTP Extension Field Types" registry.

[RFC7821]は、「NTP拡張フィールドタイプ」レジストリに新しいエントリ、チェックサム補数を追加します。

[RFC7822] clarifies the processing rules for Extension Field Types, particularly around the interaction with the Message Authentication Code (MAC) field. NTPv4 packets may contain a MAC that appears where one would expect the next extension field header.

[RFC7822]は、特にメッセージ認証コード（MAC）フィールドとの相互作用を中心に、拡張フィールドタイプの処理ルールを明確にします。NTPV4パケットには、次の拡張フィールドヘッダーが予想される場所に表示されるMacが含まれている場合があります。

[RFC8573] changes the cryptography used in the MAC field.

[RFC8573] MACフィールドで使用される暗号化を変更します。

[RFC8915] adds four new entries to the "NTP Extension Field Types" registry.

[RFC8915]は、「NTP拡張フィールドタイプ」レジストリに4つの新しいエントリを追加します。

The following problems exist with the current registry:

現在のレジストリには次の問題があります。

* Many of the entries in the "NTP Extension Field Types" registry have swapped some of the nibbles; for example, 0x0302 was listed for Cookie Message Request instead of 0x0203. The errors are due to documentation errors with the original implementation of Autokey. This document marks the erroneous values as reserved, in case there is an implementation using the registered values instead of what the original implementation used. Applications that used those values would have realized that they did not interoperate with the dominant (if not only) implementation at the time. Marking the values as reserved ensures that any such applications continue to work as is.

* 「NTP拡張フィールドタイプ」レジストリのエントリの多くは、いくつかのニブルを交換しました。たとえば、0x0203ではなく、Cookieメッセージリクエスト用に0x0302がリストされました。エラーは、Autokeyの元の実装によるドキュメントエラーによるものです。このドキュメントは、元の実装が使用したものではなく登録値を使用して実装がある場合に備えて、誤った値を予約されているとマークします。これらの値を使用したアプリケーションは、その時点での（ただし、ではないにしても）実装と相互運用していないことに気付いたでしょう。予約されている値をマークすると、そのようなアプリケーションがそのまま機能し続けることが保証されます。

* Some values were mistakenly reused.

* いくつかの値は誤って再利用されました。

2.3. Network Time Security Registries
2.3. ネットワークタイムセキュリティレジストリ

[RFC8915] defines the NTS protocol. The related registries are listed here for completeness, but there are no changes specified in this document.

[RFC8915] NTSプロトコルを定義します。関連するレジストリは完全性のためにここにリストされていますが、このドキュメントで指定された変更はありません。

In [RFC8915]:

[RFC8915]：

Sections 7.1 through 7.5 (inclusive) added entries to existing registries.

セクション7.1〜7.5（包括的）既存のレジストリにエントリを追加しました。

Section 7.6 created the "Network Time Security Key Establishment Record Types" registry that partitions the range into three different registration policies: IETF Review, Specification Required, and Private or Experimental Use.

セクション7.6は、「ネットワークタイムセキュリティキー設立レコードタイプ」を作成しました。レジストリは、範囲を3つの異なる登録ポリシーに分割します。IETFレビュー、仕様が必要、プライベートまたは実験的使用。

Section 7.7 created the "Network Time Security Next Protocols" registry that similarly partitions the range.

セクション7.7は、同様に範囲を分割する「ネットワークタイムセキュリティ次のプロトコル」レジストリを作成しました。

Section 7.8 created the "Network Time Security Error Codes" and "Network Time Security Warning Codes" registries. Both registries are partitioned the same way.

セクション7.8は、「ネットワークタイムセキュリティエラーコード」と「ネットワークタイムセキュリティ警告コード」レジストリを作成しました。どちらのレジストリも同じ方法で分割されています。

3. NTP Registry Updates
3. NTPレジストリの更新

The following general guidelines apply to the NTP registries:

NTPレジストリには、次の一般的なガイドラインが適用されます。

* A partition of the "NTP Extension Field Types" registry is reserved for Private or Experimental Use.

* 「NTP拡張フィールドタイプ」レジストリのパーティションは、プライベートまたは実験的使用のために予約されています。

* In the "NTP Reference Identifier Codes" and "NTP Kiss-o'-Death Codes" registries, entries with ASCII fields are now limited to uppercase letters or digits. Fields starting with 0x58, the uppercase letter "X", are reserved for Private or Experimental Use.

* 「NTPリファレンス識別子コード」および「NTP Kiss-O'-Deathコード」レジストリでは、ASCIIフィールドのエントリは大文字または数字に限定されています。0x58から始まるフィールド、大文字「x」は、プライベートまたは実験的使用のために予約されています。

* The policy for each registry is now Specification Required, as defined in [RFC8126], Section 4.6.

* [RFC8126]、セクション4.6で定義されているように、各レジストリのポリシーが必要になりました。
