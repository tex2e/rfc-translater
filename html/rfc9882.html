Abstract
概要

The Module-Lattice-Based Digital Signature Algorithm (ML-DSA), as defined by NIST in FIPS 204, is a post-quantum digital signature scheme that aims to be secure against an adversary in possession of a Cryptographically Relevant Quantum Computer (CRQC). This document specifies the conventions for using the ML-DSA signature algorithm with the Cryptographic Message Syntax (CMS). In addition, the algorithm identifier syntax is provided.

NIST によって FIPS 204 で定義されているモジュール格子ベースのデジタル署名アルゴリズム (ML-DSA) は、暗号関連量子コンピューター (CRQC) を所有する敵対者に対して安全であることを目的としたポスト量子デジタル署名スキームです。この文書では、暗号メッセージ構文 (CMS) で ML-DSA 署名アルゴリズムを使用するための規則を指定します。さらに、アルゴリズム識別子の構文が提供されます。

1. Introduction
1. はじめに

The Module-Lattice-Based Digital Signature Algorithm (ML-DSA) is a post-quantum digital signature algorithm standardised by the US National Institute of Standards and Technology (NIST) as part of their post-quantum cryptography standardisation process. It offers smaller signatures and significantly faster runtimes than SLH-DSA [FIPS205], an alternative post-quantum signature algorithm also standardised by NIST. This document specifies the use of ML-DSA in the CMS at three security levels: ML-DSA-44, ML-DSA-65, and ML-DSA-87. See Appendix B of [RFC9881] for more information on the security levels and key sizes of ML-DSA.

モジュール格子ベースのデジタル署名アルゴリズム (ML-DSA) は、米国国立標準技術研究所 (NIST) がポスト量子暗号化標準化プロセスの一環として標準化したポスト量子デジタル署名アルゴリズムです。これは、同様に NIST によって標準化された代替ポスト量子署名アルゴリズムである SLH-DSA [FIPS205] よりも小さい署名と大幅に高速な実行時間を提供します。この文書では、CMS での ML-DSA の使用を、ML-DSA-44、ML-DSA-65、および ML-DSA-87 の 3 つのセキュリティ レベルで指定します。ML-DSA のセキュリティ レベルと鍵サイズの詳細については、[RFC9881] の付録 B を参照してください。

Prior to standardisation, ML-DSA was known as Dilithium. ML-DSA and Dilithium are not compatible.

標準化される前は、ML-DSA はダイリチウムとして知られていました。ML-DSA とダイリチウムには互換性がありません。

For each of the ML-DSA parameter sets, an algorithm identifier OID has been specified.

ML-DSA パラメータ セットごとに、アルゴリズム識別子 OID が指定されています。

[FIPS204] also specifies a pre-hashed variant of ML-DSA, called HashML-DSA. Use of HashML-DSA in the CMS is not specified in this document. See Section 3.1 for more details.

[FIPS204] では、HashML-DSA と呼ばれる、ML-DSA の事前ハッシュされたバリアントも指定されています。CMS での HashML-DSA の使用は、このドキュメントでは指定されていません。詳細については、セクション 3.1 を参照してください。

3. Signed-Data Conventions
3. 署名付きデータの規則
3.1. Pure Mode Versus Pre-Hash Mode
3.1. 純粋モードと事前ハッシュモード

[RFC5652] specifies that digital signatures for CMS are produced using a digest of the message to be signed and the signer's private key. At the time RFC 5652 was published, all signature algorithms supported in the CMS required a message digest to be calculated externally to that algorithm, which would then be supplied to the algorithm implementation when calculating and verifying signatures. Since then, EdDSA [RFC8032], SLH-DSA [FIPS205] and ML-DSA have also been standardised, and these algorithms support both a "pure" and a "pre-hash" mode. In the pre-hash mode, a message digest (the "pre-hash") is calculated separately and supplied to the signature algorithm as described above. In the pure mode, the message to be signed or verified is instead supplied directly to the signature algorithm. When EdDSA [RFC8419] and SLH-DSA [RFC9814] are used with CMS, only the pure mode of those algorithms is specified. This is because in most situations, CMS signatures are computed over a set of signed attributes that contain a hash of the content, rather than being computed over the message content itself. Since signed attributes are typically small, use of pre-hash modes in the CMS wouldn't significantly reduce the size of the data to be signed, and hence offers no benefit. This document follows that convention and does not specify the use of ML-DSA's pre-hash mode ("HashML-DSA") in the CMS.

[RFC5652] は、CMS のデジタル署名が、署名されるメッセージのダイジェストと署名者の秘密鍵を使用して生成されることを指定しています。RFC 5652 が発行された時点では、CMS でサポートされているすべての署名アルゴリズムでは、メッセージ ダイジェストをそのアルゴリズムの外部で計算する必要があり、署名の計算と検証時にアルゴリズム実装にそのダイジェストが提供されます。それ以来、EdDSA [RFC8032]、SLH-DSA [FIPS205]、および ML-DSA も標準化され、これらのアルゴリズムは「純粋」モードと「事前ハッシュ」モードの両方をサポートしています。プレハッシュ モードでは、メッセージ ダイジェスト (「プレハッシュ」) が個別に計算され、上で説明したように署名アルゴリズムに供給されます。純粋モードでは、署名または検証されるメッセージが署名アルゴリズムに直接供給されます。EdDSA [RFC8419] および SLH-DSA [RFC9814] が CMS で使用される場合、それらのアルゴリズムの純粋モードのみが指定されます。これは、ほとんどの状況で、CMS 署名がメッセージ コンテンツ自体に対して計算されるのではなく、コンテンツのハッシュを含む一連の署名付き属性に対して計算されるためです。署名された属性は通常小さいため、CMS で事前ハッシュ モードを使用しても、署名されるデータのサイズは大幅に削減されず、メリットがありません。このドキュメントはその規則に従い、CMS での ML-DSA の事前ハッシュ モード (「HashML-DSA」) の使用を指定しません。

3.2. Signature Generation and Verification
3.2. 署名の生成と検証

[RFC5652] describes the two methods that are used to calculate and verify signatures in the CMS. One method is used when signed attributes are present in the signedAttrs field of the relevant SignerInfo, and another is used when signed attributes are absent. Each method produces a different "message digest" to be supplied to the signature algorithm in question, but because the pure mode of ML-DSA is used, the "message digest" is in fact the entire message. Use of signed attributes is preferred, but the conventions for signed-data without signed attributes is also described below for completeness.

[RFC5652] では、CMS で署名を計算および検証するために使用される 2 つの方法について説明しています。1 つのメソッドは、関連する SignerInfo の signedAttrs フィールドに署名された属性が存在する場合に使用され、もう 1 つのメソッドは、署名された属性が存在しない場合に使用されます。各メソッドは、対象の署名アルゴリズムに提供される異なる「メッセージ ダイジェスト」を生成しますが、ML-DSA の純粋モードが使用されるため、「メッセージ ダイジェスト」は実際にはメッセージ全体です。署名付き属性の使用が推奨されますが、完全を期すために、署名付き属性を持たない署名付きデータの規則についても以下で説明します。

When signed attributes are absent, ML-DSA (pure mode) signatures are computed over the content of the signed-data. As described in Section 5.4 of [RFC5652], the "content" of a signed-data is the value of the encapContentInfo eContent OCTET STRING. The tag and length octets are not included.

署名付き属性が存在しない場合、ML-DSA (純粋モード) 署名は署名データの内容に対して計算されます。[RFC5652] のセクション 5.4 で説明されているように、署名付きデータの「コンテンツ」は encapContentInfo eContent OCTET STRING の値です。タグと長さのオクテットは含まれません。

When signed attributes are included, ML-DSA (pure mode) signatures are computed over the complete DER encoding of the SignedAttrs value contained in the SignerInfo's signedAttrs field. As described in Section 5.4 of [RFC5652], this encoding includes the tag and length octets, but an EXPLICIT SET OF tag is used rather than the IMPLICIT [0] tag that appears in the final message. At a minimum, the signedAttrs field MUST include a content-type attribute and a message-digest attribute. The message-digest attribute contains a hash of the content of the signed-data, where the content is as described for the absent signed attributes case above. Recalculation of the hash value by the recipient is an important step in signature verification.

署名付き属性が含まれる場合、ML-DSA (ピュア モード) 署名は、SignerInfo の signedAttrs フィールドに含まれる SignedAttrs 値の完全な DER エンコーディングに基づいて計算されます。[RFC5652] のセクション 5.4 で説明されているように、このエンコーディングにはタグと長さのオクテットが含まれますが、最終メッセージに表示される IMPLICIT [0] タグではなく、EXPLICIT SET OF タグが使用されます。少なくとも、signedAttrs フィールドには content-type 属性と message-digest 属性が含まれなければなりません (MUST)。メッセージ ダイジェスト属性には署名データのコンテンツのハッシュが含まれており、その内容は上記の署名属性が存在しない場合で説明したとおりです。受信者によるハッシュ値の再計算は、署名検証における重要なステップです。

Section 4 of [RFC9814] describes how, when the content of a signed-data is large, performance may be improved by including signed attributes. This is as true for ML-DSA as it is for SLH-DSA, although ML-DSA signature generation and verification is significantly faster than SLH-DSA.

[RFC9814] のセクション 4 では、署名付きデータの内容が大きい場合に、署名付き属性を含めることでパフォーマンスを向上させる方法について説明しています。これは SLH-DSA と同様に ML-DSA にも当てはまりますが、ML-DSA 署名の生成と検証は SLH-DSA よりも大幅に高速です。

ML-DSA has a context string input that can be used to ensure that different signatures are generated for different application contexts. When using ML-DSA as specified in this document, the context string is set to the empty string.

ML-DSA には、さまざまなアプリケーション コンテキストに対してさまざまな署名が生成されるようにするために使用できるコンテキスト文字列入力があります。このドキュメントで指定されているように ML-DSA を使用する場合、コンテキスト文字列は空の文字列に設定されます。

3.3. SignerInfo Content
3.3. 署名者情報のコンテンツ

When using ML-DSA, the fields of a SignerInfo are used as follows:

ML-DSA を使用する場合、SignerInfo のフィールドは次のように使用されます。

digestAlgorithm:

ダイジェストアルゴリズム:

Per Section 5.3 of [RFC5652], the digestAlgorithm field identifies the message digest algorithm used by the signer and any associated parameters. Each ML-DSA parameter set has a collision strength parameter, represented by the "λ" (GREEK SMALL LETTER LAMDA, U+03BB) symbol in [FIPS204]. When signers utilise signed attributes, their choice of digest algorithm may impact the overall security level of their signature. Selecting a digest algorithm that offers λ bits of security strength against second preimage attacks and collision attacks is sufficient to meet the security level offered by a given parameter set, so long as the digest algorithm produces at least 2 * λ bits of output. The overall security strength offered by an ML-DSA signature calculated over signed attributes is constrained by either the digest algorithm's strength or the strength of the ML-DSA parameter set, whichever is lower. Verifiers MAY reject a signature if the signer's choice of digest algorithm does not meet the security requirements of their choice of ML-DSA parameter set. Table 1 shows appropriate SHA-2 and SHA-3 digest algorithms for each parameter set.

[RFC5652] のセクション 5.3 に従って、digestAlgorithm フィールドは、署名者が使用するメッセージ ダイジェスト アルゴリズムおよび関連パラメータを識別します。各 ML-DSA パラメータ セットには衝突強度パラメータがあり、[FIPS204] では「λ」 (ギリシャ小文字 LAMDA、U+03BB) 記号で表されます。署名者が署名付き属性を利用する場合、ダイジェスト アルゴリズムの選択が署名の全体的なセキュリティ レベルに影響を与える可能性があります。ダイジェスト アルゴリズムが少なくとも 2 * λ ビットの出力を生成する限り、2 番目のプリイメージ攻撃および衝突攻撃に対して λ ビットのセキュリティ強度を提供するダイジェスト アルゴリズムを選択することで、特定のパラメーター セットによって提供されるセキュリティ レベルを満たすのに十分です。署名された属性に対して計算された ML-DSA 署名によって提供される全体的なセキュリティ強度は、ダイジェスト アルゴリズムの強度または ML-DSA パラメータ セットの強度のいずれか低い方によって制限されます。検証者は、署名者の選択したダイジェストアルゴリズムが、選択した ML-DSA パラメータセットのセキュリティ要件を満たさない場合、署名を拒否してもよい(MAY)。表 1 は、各パラメーター セットに適切な SHA-2 および SHA-3 ダイジェスト アルゴリズムを示しています。

SHA-512 [FIPS180] MUST be supported for use with the variants of ML-DSA in this document. SHA-512 is suitable for all ML-DSA parameter sets and provides an interoperable option for legacy CMS implementations that wish to migrate to use post-quantum cryptography, but that may not support use of SHA-3 derivatives at the CMS layer. However, other hash functions MAY also be supported; in particular, SHAKE256 SHOULD be supported, as this is the digest algorithm used internally in ML-DSA. When SHA-512 is used, the id-sha512 [RFC5754] digest algorithm identifier is used and the parameters field MUST be omitted. When SHAKE256 is used, the id-shake256 [RFC8702] digest algorithm identifier is used and the parameters field MUST be omitted. SHAKE256 produces 512 bits of output when used as a message digest algorithm in the CMS.

SHA-512 [FIPS180] このドキュメントの ML-DSA のバリアントでの使用はサポートされなければなりません (MUST)。SHA-512 はすべての ML-DSA パラメータ セットに適しており、ポスト量子暗号を使用するように移行したいが、CMS 層での SHA-3 派生の使用をサポートしていない可能性がある従来の CMS 実装に相互運用可能なオプションを提供します。ただし、他のハッシュ関数もサポートされてもよい(MAY)。特に、SHAKE256 は、ML-DSA で内部的に使用されるダイジェスト アルゴリズムであるため、サポートされるべきです。SHA-512 が使用される場合、id-sha512 [RFC5754] ダイジェスト アルゴリズム識別子が使用され、パラメーター フィールドは省略されなければなりません (MUST)。SHAKE256 が使用される場合、id-shake256 [RFC8702] ダイジェスト アルゴリズム識別子が使用され、パラメーター フィールドは省略されなければなりません (MUST)。SHAKE256 は、CMS でメッセージ ダイジェスト アルゴリズムとして使用される場合、512 ビットの出力を生成します。

When signing using ML-DSA without including signed attributes, the algorithm specified in the digestAlgorithm field has no meaning, as ML-DSA computes signatures over entire messages rather than externally computed digests. As such, the considerations above and in Table 1 do not apply. Nonetheless, in this case implementations MUST specify SHA-512 as the digestAlgorithm in order to minimise the likelihood of an interoperability failure. 4. Security Considerations
4. セキュリティに関する考慮事項

The security considerations in [RFC5652] and [RFC9881] apply to this specification.

[RFC5652] および [RFC9881] のセキュリティに関する考慮事項がこの仕様に適用されます。

Security of the ML-DSA private key is critical. Compromise of the private key will enable an adversary to forge arbitrary signatures.

ML-DSA 秘密キーのセキュリティは非常に重要です。秘密キーが侵害されると、敵対者が任意の署名を偽造できるようになります。

ML-DSA depends on high-quality random numbers that are suitable for use in cryptography. The use of inadequate pseudo-random number generators (PRNGs) to generate such values can significantly undermine the security properties offered by a cryptographic algorithm. For instance, an attacker may find it much easier to reproduce the PRNG environment that produced any private keys, searching the resulting small set of possibilities, rather than brute-force searching the whole key space. The generation of random numbers of a sufficient level of quality for use in cryptography is difficult; see Section 3.6.1 of [FIPS204] for some additional information.

ML-DSA は、暗号化での使用に適した高品質の乱数に依存しています。このような値を生成するために不適切な擬似乱数ジェネレーター (PRNG) を使用すると、暗号化アルゴリズムによって提供されるセキュリティー特性が大幅に損なわれる可能性があります。たとえば、攻撃者は、キー空間全体をブルートフォースで検索するよりも、秘密キーを生成した PRNG 環境を再現し、結果として得られる小さな可能性のセットを検索する方がはるかに簡単であると考える可能性があります。暗号化で使用するのに十分な品質の乱数を生成することは困難です。追加情報については、[FIPS204] のセクション 3.6.1 を参照してください。

By default, ML-DSA signature generation uses randomness from two sources: fresh random data generated during signature generation, and precomputed random data included in the signer's private key. This is referred to as the "hedged" variant of ML-DSA. Inclusion of both sources of random data can help mitigate against faulty random number generators, side-channel attacks, and fault attacks. [FIPS204] also permits creating deterministic signatures using just the precomputed random data in the signer's private key. The same verification algorithm is used to verify both hedged and deterministic signatures, so this choice does not affect interoperability. The signer SHOULD NOT use the deterministic variant of ML-DSA on platforms where side-channel attacks or fault attacks are a concern. Side-channel attacks and fault attacks against ML-DSA are an active area of research [WNGD2023] [KPLG2024]. Future protection against these styles of attack may involve interoperable changes to the implementation of ML-DSA's internal functions. Implementers SHOULD consider implementing such protection measures if it would be beneficial for their particular use cases.

デフォルトでは、ML-DSA 署名の生成では、署名生成中に生成された新しいランダム データと、署名者の秘密キーに含まれる事前計算されたランダム データという 2 つのソースからのランダム性が使用されます。これは、ML-DSA の「ヘッジ」バリアントと呼ばれます。ランダム データの両方のソースを含めることで、障害のある乱数ジェネレーター、サイドチャネル攻撃、および障害攻撃を軽減できます。[FIPS204] では、署名者の秘密鍵内の事前計算されたランダム データのみを使用して決定論的な署名を作成することも許可されています。ヘッジ署名と決定論的署名の両方を検証するために同じ検証アルゴリズムが使用されるため、この選択は相互運用性に影響しません。署名者は、サイドチャネル攻撃や障害攻撃が懸念されるプラットフォームでは、ML-DSA の決定論的バリアントを使用すべきではありません (SHOULD NOT)。ML-DSA に対するサイドチャネル攻撃とフォールト攻撃は、活発な研究分野です [WNGD2023] [KPLG2024]。これらのスタイルの攻撃に対する将来の保護には、ML-DSA の内部機能の実装に対する相互運用可能な変更が含まれる可能性があります。実装者は、特定のユースケースにとって有益である場合、そのような保護手段の実装を検討すべきです(SHOULD)。

To avoid algorithm substitution attacks, the CMSAlgorithmProtection attribute defined in [RFC6211] SHOULD be included in signed attributes.

アルゴリズム置換攻撃を回避するには、[RFC6211] で定義されている CMSAlgorithmProtection 属性を署名付き属性に含める必要があります (SHOULD)。

5. Operational Considerations
5. 運用上の考慮事項

If ML-DSA signing is implemented in a hardware device such as a hardware security module (HSM) or a portable cryptographic token, implementers might want to avoid sending the full content to the device for performance reasons. By including signed attributes, which necessarily includes the message-digest attribute and the content-type attribute as described in Section 5.3 of [RFC5652], the much smaller set of signed attributes are sent to the device for signing.

ML-DSA 署名がハードウェア セキュリティ モジュール (HSM) やポータブル暗号トークンなどのハードウェア デバイスに実装されている場合、実装者はパフォーマンス上の理由から、完全なコンテンツをデバイスに送信することを避けたい場合があります。[RFC5652] のセクション 5.3 で説明されているように、必ず message-digest 属性と content-type 属性を含む署名付き属性を含めることにより、はるかに小さい署名付き属性のセットが署名のためにデバイスに送信されます。

Additionally, the pure variant of ML-DSA does support a form of pre-hash via external calculation of the "μ" (GREEK SMALL LETTER MU, U+03BC) "message representative" value described in Section 6.2 of [FIPS204]. This value may "optionally be computed in a different cryptographic module" and supplied to the hardware device, rather than requiring the entire message to be transmitted. Appendix D of [RFC9881] describes use of external μ calculations in further detail.

さらに、ML-DSA の純粋なバリアントは、[FIPS204] のセクション 6.2 で説明されている「μ」 (ギリシャの小文字 MU、U+03BC) 「メッセージ代表」値の外部計算による事前ハッシュの形式をサポートします。この値は、メッセージ全体を送信する必要はなく、「オプションで別の暗号化モジュールで計算」され、ハードウェア デバイスに供給される場合があります。[RFC9881] の付録 D では、外部μ計算の使用法についてさらに詳しく説明しています。
