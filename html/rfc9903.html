1. Overview
1. 概要

This document defines a YANG data model [RFC7950] that can be used to manage OSPFv2 extensions for Segment Routing [RFC2328] [RFC8665] and OSPFv3 extensions for Segment Routing [RFC5340] [RFC8666] over the MPLS data plane. The defined YANG data model is an augmentation to the OSPF YANG data model [RFC9129].

この文書は、MPLS データ プレーン上でセグメント ルーティングの OSPFv2 拡張 [RFC2328] [RFC8665] およびセグメント ルーティングの OSPFv3 拡張 [RFC5340] [RFC8666] を管理するために使用できる YANG データ モデル [RFC7950] を定義します。定義された YANG データ モデルは、OSPF YANG データ モデル [RFC9129] を拡張したものです。

2. Design of the YANG Module for OSPF MPLS Segment Routing
2. OSPF MPLS セグメント ルーティング用の YANG モジュールの設計

The YANG data model in this document consists of a single YANG module, "ietf-ospf-sr-mpls". The "ietf-ospf-sr-mpls" YANG module includes OSPF Segment Routing extensions for the MPLS data plane for both OSPFv2 [RFC8665] and OSPFv3 [RFC8666].

このドキュメントの YANG データ モデルは、単一の YANG モジュール「ietf-ospf-sr-mpls」で構成されています。「ietf-ospf-sr-mpls」YANG モジュールには、OSPFv2 [RFC8665] と OSPFv3 [RFC8666] の両方の MPLS データ プレーン用の OSPF セグメント ルーティング拡張機能が含まれています。

The "ietf-ospf-sr-mpls" YANG module requires support for the base Segment Routing module [RFC9020], which defines the global Segment Routing configuration independent of any specific routing protocol configuration, and support of the OSPF base model [RFC9129], which defines the basic OSPF configuration and state.

「ietf-ospf-sr-mpls」YANG モジュールは、特定のルーティング プロトコル設定に依存しないグローバル セグメント ルーティング設定を定義する基本セグメント ルーティング モジュール [RFC9020] のサポートと、基本的な OSPF 設定と状態を定義する OSPF 基本モデル [RFC9129] のサポートを必要とします。

The "ietf-ospf-sr-mpls" module defines both the data nodes to configure OSPF Segment Routing MPLS extensions and the additions to OSPF Link State Advertisements (LSAs) necessary to support Segment Routing over MPLS (SR-MPLS). The OSPF configuration includes:

「ietf-ospf-sr-mpls」モジュールは、OSPF セグメント ルーティング MPLS 拡張を設定するためのデータ ノードと、セグメント ルーティング オーバー MPLS（SR-MPLS）をサポートするために必要な OSPF リンク ステート アドバタイズメント（LSA）への追加の両方を定義します。OSPF 構成には以下が含まれます。

* OSPF instance level configuration imported from the "ietf-segment-routing-mpls" YANG module, including the mapping server bindings and the per-protocol Segment Routing Global Block (SRGB) (refer to the "sr-control-plane" grouping [RFC9020]).

* 「ietf-segment-routing-mpls」YANG モジュールからインポートされた OSPF インスタンス レベルの設定。マッピング サーバー バインディングおよびプロトコルごとのセグメント ルーティング グローバル ブロック (SRGB) が含まれます (「sr-control-plane」グループ化 [RFC9020] を参照)。

* OSPF area level configuration, which enables SR-MPLS on all interfaces as well as advertisement of SR-MPLS information in Link State Advertisements (LSAs).

* OSPF エリア レベルの設定。すべてのインターフェイスで SR-MPLS を有効にし、リンク ステート アドバタイズメント (LSA) で SR-MPLS 情報のアドバタイズメントを有効にします。

* OSPF interface level configuration for Adjacency Segment Identifiers (Adj-SIDs) corresponding to specific neighbors on multi-access interfaces (OSPF interface type broadcast or Non-Broadcast Multi-Access (NBMA)). The adjacency-sid definitions are imported from the "ietf-segment-routing-mpls" YANG module (refer to the "igp-interface" grouping [RFC9020]).

* マルチアクセス インターフェイス（OSPF インターフェイス タイプ ブロードキャストまたは非ブロードキャスト マルチアクセス（NBMA））上の特定のネイバーに対応する隣接セグメント識別子（Adj-SID）の OSPF インターフェイス レベルの設定。adjacency-sid 定義は、「ietf-segment-routing-mpls」YANG モジュールからインポートされます (「igp-interface」グループ [RFC9020] を参照)。

* OSPF interface level configuration for Topology Independent Loop-Free Alternative (TI-LFA) using the MPLS data plane. TI-LFA is described in [RFC9855].

* MPLS データ プレーンを使用した、Topology Independent Loop-Free Alternative (TI-LFA) の OSPF インターフェイス レベルの設定。TI-LFA は [RFC9855] で説明されています。

The operational state (read-only) additions specific to OSPFv2 LSA include:

OSPFv2 LSA に固有の動作状態 (読み取り専用) の追加には次のものがあります。

* OSPFv2 Extended Prefix Range TLV encodings [RFC8665] in the OSPF Extended Prefix Opaque LSA [RFC7684].

* OSPFv2 拡張プレフィックス範囲 TLV エンコーディング [RFC8665]、OSPF 拡張プレフィックス不透明 LSA [RFC7684]。

* OSPFv2 Prefix-SID Sub-TLV encodings [RFC8665], including the OSPF Extended Prefix TLV, which is advertised in the OSPF Extended Prefix Opaque LSA [RFC7684].

* OSPFv2 プレフィックス SID サブ TLV エンコーディング [RFC8665]。これには、OSPF 拡張プレフィックス Opaque LSA [RFC7684] でアドバタイズされる OSPF 拡張プレフィックス TLV が含まれます。

* OSPFv2 SR-Algorithm, SID/Label Range TLV, SR Local Block TLV, and Segment Routing Mapping Server (SRMS) Preference TLV [RFC8665] advertised in the OSPFv2 Router Information Opaque LSA [RFC7770].

* OSPFv2 SR アルゴリズム、SID/ラベル範囲 TLV、SR ローカル ブロック TLV、およびセグメント ルーティング マッピング サーバー (SRMS) 優先 TLV [RFC8665] は、OSPFv2 ルーター情報不透明 LSA [RFC7770] でアドバタイズされます。

The operational state (read-only) additions specific to OSPFv3 LSA include:

OSPFv3 LSA に固有の動作状態 (読み取り専用) の追加には次のものがあります。

* OSPFv3 Extended Prefix Range TLV encodings [RFC8666] in the OSPFv3 E-Intra-Area-Prefix-LSA, E-Inter-Area-Prefix-LSA, E-AS-External-LSA, and E-Type-7-LSA [RFC8362].

* OSPFv3 E-Intra-Area-Prefix-LSA、E-Inter-Area-Prefix-LSA、E-AS-External-LSA、および E-Type-7-LSA [RFC8362] の OSPFv3 Extended Prefix Range TLV エンコーディング [RFC8666]。

* OSPFv3 Prefix-SID Sub-TLV encodings [RFC8666] in the OSPFv3 Intra-Area-Prefix TLV, Inter-Area-Prefix TLV, External-Prefix TLV [RFC8362], and OSPFv3 Extended Prefix Range TLV [RFC8666].

* OSPFv3 エリア内プレフィックス TLV、エリア間プレフィックス TLV、外部プレフィックス TLV [RFC8362]、および OSPFv3 拡張プレフィックス範囲 TLV [RFC8666] の OSPFv3 プレフィックス SID サブ TLV エンコーディング [RFC8666]。

* OSPFv3 Adj-SID Sub-TLV [RFC8666] in the OSPFv3 Router-Link TLV [RFC8362].

* OSPFv3 Router-Link TLV [RFC8362] 内の OSPFv3 Adj-SID Sub-TLV [RFC8666]。

* OSPFv3 LAN Adj-SID Sub-TLV [RFC8666] in the OSPFv3 Router-Link TLV [RFC8362].

* OSPFv3 Router-Link TLV [RFC8362] 内の OSPFv3 LAN Adj-SID Sub-TLV [RFC8666]。

* OSPFv3 SR-Algorithm, SID/Label Range TLV, SR Local Block TLV, and SRMS Preference TLV [RFC8666] advertised in the OSPFv3 Router Information LSA [RFC7770].

* OSPFv3 SR-Algorithm、SID/Label Range TLV、SR Local Block TLV、および SRMS Preference TLV [RFC8666] は、OSPFv3 Router Information LSA [RFC7770] でアドバタイズされます。

4. Security Considerations
4. セキュリティに関する考慮事項

This section is modeled after the template described in Section 3.7 of [YANG-GUIDE].

このセクションは、[YANG-GUIDE] のセクション 3.7 で説明されているテンプレートをモデルにしています。

The "ietf-ospf-sr-mpls" YANG module defines a data model that is designed to be accessed via YANG-based management protocols, such as NETCONF [RFC6241] and RESTCONF [RFC8040]. These YANG-based management protocols (1) have to use a secure transport layer (e.g., SSH [RFC4252], TLS [RFC8446], and QUIC [RFC9000]) and (2) have to use mutual authentication.

「ietf-ospf-sr-mpls」YANG モジュールは、NETCONF [RFC6241] や RESTCONF [RFC8040] などの YANG ベースの管理プロトコル経由でアクセスできるように設計されたデータ モデルを定義します。これらの YANG ベースの管理プロトコルは、(1) 安全なトランスポート層 (SSH [RFC4252]、TLS [RFC8446]、QUIC [RFC9000] など) を使用する必要があり、(2) 相互認証を使用する必要があります。

The Network Configuration Access Control Model (NACM) [RFC8341] provides the means to restrict access for particular NETCONF or RESTCONF users to a preconfigured subset of all available NETCONF or RESTCONF protocol operations and content.

Network Configuration Access Control Model (NACM) [RFC8341] は、特定の NETCONF または RESTCONF ユーザーのアクセスを、利用可能なすべての NETCONF または RESTCONF プロトコルの操作およびコンテンツの事前構成されたサブセットに制限する手段を提供します。

There are a number of data nodes defined in this YANG module that are writable/creatable/deletable (i.e., "config true", which is the default). All writable data nodes are likely to be sensitive or vulnerable in some network environments. Write operations (e.g., edit-config) and delete operations to these data nodes without proper protection or authentication can have a negative effect on network operations. The following subtrees and data nodes have particular sensitivities/vulnerabilities:

この YANG モジュールには、書き込み可能/作成可能/削除可能 (つまり、デフォルトの「config true」) のデータ ノードが多数定義されています。一部のネットワーク環境では、すべての書き込み可能なデータ ノードが機密性が高いか脆弱である可能性があります。適切な保護や認証を行わずにこれらのデータ ノードへの書き込み操作 (edit-config など) や削除操作を行うと、ネットワーク操作に悪影響を及ぼす可能性があります。次のサブツリーとデータ ノードには、特定の感度/脆弱性があります。

* /ospf:ospf/segment-routing/enabled - Modification to the enablement for SR could result in a Denial-of-Service (DoS) attack. If an attacker disables SR, it will cause traffic disruption.

* /ospf:ospf/segment-routing/enabled - SR の有効化を変更すると、サービス拒否 (DoS) 攻撃が発生する可能性があります。攻撃者が SR を無効にすると、トラフィックが中断されます。

* /ospf:ospf/segment-routing/bindings - Modification to the local bindings could result in a DoS attack.

* /ospf:ospf/segment-routing/bindings - ローカル バインディングを変更すると、DoS 攻撃が発生する可能性があります。

* /ospf:ospf/protocol-srgb - Modification of the protocol SRGB could be used to mount a DoS attack. For example, if the protocol SRGB size is reduced to a very small value, a lot of existing segments could no longer be installed, leading to a traffic disruption.

* /ospf:ospf/protocol-srgb - プロトコル SRGB の変更は、DoS 攻撃を仕掛けるために使用される可能性があります。たとえば、プロトコルの SRGB サイズが非常に小さい値に削減されると、多くの既存のセグメントをインストールできなくなり、トラフィックの中断が発生します。

* /ospf:interfaces/ospf:interface/segment-routing - Modification of the Adj-SID could be used to mount a DoS attack. Change of an Adj-SID could be used to redirect traffic.

* /ospf:interfaces/ospf:interface/segment-routing - Adj-SID の変更は、DoS 攻撃を仕掛けるために使用される可能性があります。Adj-SID の変更は、トラフィックのリダイレクトに使用される可能性があります。

* /ospf:interfaces/ospf:interface/ospf:fast-reroute/ti-lfa - Modification of the TI-LFA enablement could lead to traffic disruption.

* /ospf:interfaces/ospf:interface/ospf:fast-reroute/ti-lfa - TI-LFA 有効化の変更により、トラフィックの中断が発生する可能性があります。

Some of the readable data nodes in this YANG module may be considered sensitive or vulnerable in some network environments. It is thus important to control read access (e.g., via get, get-config, or notification) to these data nodes. Specifically, the following subtrees and data nodes have particular sensitivities/ vulnerabilities:

この YANG モジュールの読み取り可能なデータ ノードの一部は、一部のネットワーク環境では機密であるか脆弱であると見なされる場合があります。したがって、これらのデータ ノードへの読み取りアクセス (get、get-config、通知など) を制御することが重要です。具体的には、次のサブツリーとデータ ノードには特定の感度/脆弱性があります。

* The "ietf-ospf-sr-mpls" module augments the base OSPF module Link State Database (LSDB) with various TLVs. Knowledge of these data nodes can be used to attack other routers in the OSPF domain. These attacks are documented in [RFC9129].

* 「ietf-ospf-sr-mpls」モジュールは、ベース OSPF モジュールのリンク ステート データベース (LSDB) をさまざまな TLV で拡張します。これらのデータ ノードの知識は、OSPF ドメイン内の他のルーターを攻撃するために使用される可能性があります。これらの攻撃は [RFC9129] に文書化されています。

* Knowledge of the Prefix and Adjacency SIDs advertised in LSAs could facilitate a targeted attack on the data plane by knowing the topology and path for MPLS segment-routed packets. Examples include the OSPFv2 extended-prefix-opaque-LSA/prefix-sid-sub-tlvs and OSPFv3 e-router-tlvs/link-tlv/sub-tlvs/ospfv3-lan-adj-sid-sub-tlvs.

* LSA でアドバタイズされるプレフィックスと隣接 SID を知ることで、MPLS セグメント ルーティングされたパケットのトポロジとパスを知ることにより、データ プレーンに対する標的型攻撃が容易になる可能性があります。例には、OSPFv2 extend-prefix-opaque-LSA/prefix-sid-sub-tlvs および OSPFv3 e-router-tlvs/link-tlv/sub-tlvs/ospfv3-lan-adj-sid-sub-tlvs が含まれます。

* Knowledge of Prefix and Adjacency SIDs may also divulge a view of the Segment Routing network topology architecture, which may be considered a proprietary asset.

* プレフィックスおよび隣接 SID の知識により、独自の資産とみなされるセグメント ルーティング ネットワーク トポロジ アーキテクチャの見解が明らかになる場合もあります。

* Knowledge of the advertised SID/Label Range TLVs, SR Local Block TLVs, and SRMS TLVs may facilitate a DoS attack on the advertising node's SR and SRMS functionality. Examples include augmentations to the OSPFv2 ri-opaque/sid-range-tlvs and OSPFv3 router-information/local-block-tlvs.

* アドバタイズされた SID/ラベル範囲 TLV、SR ローカル ブロック TLV、および SRMS TLV を知ると、アドバタイジング ノードの SR および SRMS 機能に対する DoS 攻撃が容易になる可能性があります。例には、OSPFv2 ri-opaque/sid-range-tlvs および OSPFv3 router-information/local-block-tlvs の拡張が含まれます。

There are no particularly sensitive RPC or action operations.

特に機密性の高い RPC 操作やアクション操作はありません。

この YANG モジュールは、ネットワーク環境で機密または脆弱であると考えられるノードを定義する他の YANG モジュールのグループを使用します。ネットワーク環境においてどのノードが機密であるか脆弱であるとみなされるかについては、[RFC9020] のセキュリティに関する考慮事項を参照してください。

Appendix A. A Configuration Example

The following is an XML example using the "ietf-ospf-sr-mpls" YANG module and [RFC9020].

以下は、「ietf-ospf-sr-mpls」YANG モジュールと [RFC9020] を使用した XML の例です。

Note: '\' line wrapping per [RFC8792].

注: [RFC8792] による「\」行の折り返し。 

   <?xml version='1.0' encoding='UTF-8'?>
     <routing xmlns="urn:ietf:params:xml:ns:yang:ietf-routing">
       <router-id>1.1.1.1</router-id>
       <control-plane-protocols>
         <control-plane-protocol>
           <type xmlns:ospf="urn:ietf:params:xml:ns:yang:ietf-ospf">\
             ospf:ospfv2</type>
           <name>OSPFv2</name>
           <ospf xmlns="urn:ietf:params:xml:ns:yang:ietf-ospf">
             <areas>
               <area>
                 <area-id>0.0.0.0</area-id>
                 <segment-routing xmlns="urn:ietf:params:xml:ns:yang:\
                   ietf-ospf-sr-mpls"></segment-routing>
                 <interfaces>
                   <interface>
                     <name>eth0</name>
                     <segment-routing xmlns="urn:ietf:params:xml:ns:\
                       yang:ietf-ospf-sr-mpls">
                       <adjacency-sid>
                         <adj-sids>
                           <value>3888</value>
                         </adj-sids>
                       </adjacency-sid>
                     </segment-routing>
                   </interface>
                 </interfaces>
               </area>
             </areas>
             <segment-routing xmlns="urn:ietf:params:xml:ns:yang:\
               ietf-ospf-sr-mpls">
               <enabled>true</enabled>
             </segment-routing>
             <protocol-srgb xmlns="urn:ietf:params:xml:ns:yang:\
               ietf-ospf-sr-mpls">
               <srgb>
                 <lower-bound>4000</lower-bound>
                 <upper-bound>5000</upper-bound>
               </srgb>
             </protocol-srgb>
           </ospf>
         </control-plane-protocol>
       </control-plane-protocols>
     </routing>
