Abstract
概要

This document defines two CBOR Object Signing and Encryption (COSE) header parameters for incorporating timestamping based on RFC 3161 into COSE message structures (COSE_Sign and COSE_Sign1). This enables the use of established timestamping infrastructure per RFC 3161 in COSE-based protocols.

1. Introduction
1. はじめに

RFC 3161 [RFC3161] provides a method for timestamping a message digest to prove that it was created before a given time.

RFC 3161 [RFC3161] は、メッセージ ダイジェストが特定の時刻より前に作成されたことを証明するために、メッセージ ダイジェストにタイムスタンプを付ける方法を提供します。

This document defines two new CBOR Object Signing and Encryption (COSE) [RFC9052] header parameters that carry the TimeStampToken (TST) output [RFC3161], thus allowing existing and widely deployed trust infrastructure to be used with COSE structures used for signing (COSE_Sign and COSE_Sign1).

この文書は、TimeStampToken (TST) 出力 [RFC3161] を伝送する 2 つの新しい CBOR Object Signing and Encryption (COSE) [RFC9052] ヘッダー パラメーターを定義します。これにより、広く展開されている既存の信頼インフラストラクチャを、署名に使用される COSE 構造 (COSE_Sign および COSE_Sign1) とともに使用できるようになります。

1.1. Use Cases
1.1. 使用例

This section discusses two use cases, each representing one of the two modes of use defined in Section 2. As the security characteristics of the two cases differ, care must be taken when choosing the appropriate mode for a given application. See Section 5.1 for a discussion on the security of the implementations.

このセクションでは、セクション 2 で定義した 2 つの使用モードの 1 つを表す 2 つの使用例について説明します。2 つの使用例のセキュリティ特性は異なるため、特定のアプリケーションに適切なモードを選択するときは注意が必要です。実装のセキュリティについては、セクション 5.1 を参照してください。

The primary use case is that of "long-term signatures", i.e., signatures that can still be verified even after the signing certificate has expired. This can address situations where it is important to prevent subsequent denial by the signer or to verify signatures made using (very) short-term certificates. To achieve this, the document signer acquires a fresh TST for the document's signature from a trusted Time Stamping Authority (TSA) [RFC3161] and concatenates it with the document. Later, when a relying party verifies the signed document and its associated TST, they can be certain that the document was signed _at least_ at the time specified by the TSA and that the signing certificate was valid at the time the signature was made.

主な使用例は、「長期署名」、つまり、署名証明書の有効期限が切れた後でもまだ検証できる署名です。これにより、署名者によるその後の拒否を防止したり、(非常に) 短期間の証明書を使用して作成された署名を検証したりすることが重要な状況に対処できます。これを実現するために、文書の署名者は、信頼できるタイムスタンプ局 (TSA) [RFC3161] から文書の署名用の新しい TST を取得し、それを文書と連結します。その後、信頼当事者が署名された文書とそれに関連する TST を検証すると、その文書が少なくとも TSA によって指定された時刻に署名され、署名証明書が署名作成時に有効であったことを確認できます。

This primary usage scenario motivates the "COSE, Then Timestamp" mode described in Section 2.1.

この主な使用シナリオは、セクション 2.1 で説明されている「COSE、次にタイムスタンプ」モードを動機付けます。

The second use case is new. It is the notarization of a signed document by registering it with a transparency service. This is common practice for ensuring the accountability and auditability of issued documents, which are typically referred to as "statements" in this context. It is also common practice to only register the signed parts of a statement (the "signed statement" portion) with a transparency service, in order to reduce the complexity of consistency checks at a later stage and to avoid the need to retrieve or reconstruct unsigned parts. Once the signed parts of a document have been registered in the append-only log at a transparency service, the log entry cannot be changed. In order to avoid losing the TST during the registration process, the TST must be included in the signed statement. To achieve this, the issuer acquires a TST from a TSA, includes it in the to-be-signed part of the statement so that the resulting signed statement includes the TST, and then registers the signed parts (rendering it a "transparent statement"). Later on, a relying party consuming the transparent statement including the TST can be certain that the statement was signed by the issuer _at least_ at the time specified by the TSA. If the issuer's signing key has expired (or has been compromised), the authenticity of the statement can be ascertained by ensuring that no revocation information was made public before the time asserted by the issuer and registered at the transparency service.

2 番目の使用例は新しいものです。これは、署名された文書を透明性サービスに登録することによって公証することです。これは、発行された文書 (この文脈では通常「声明」と呼ばれます) の説明責任と監査可能性を確保するための一般的な慣行です。また、後の段階での一貫性チェックの複雑さを軽減し、署名されていない部分を取得または再構築する必要性を避けるために、ステートメントの署名された部分 (「署名されたステートメント」部分) のみを透過性サービスに登録することも一般的です。文書の署名部分が透過性サービスの追加専用ログに登録されると、ログ エントリを変更することはできません。登録プロセス中に TST が失われないようにするには、署名されたステートメントに TST を含める必要があります。これを実現するために、発行者は TSA から TST を取得し、それをステートメントの署名対象部分に含めて、結果として得られる署名付きステートメントに TST が含まれるようにしてから、署名部分を登録します (これを「透過的ステートメント」とします)。その後、TST を含む透過的ステートメントを利用する依存当事者は、そのステートメントが「少なくとも」TSA によって指定された時刻に発行者によって署名されたことを確認できます。発行者の署名キーの有効期限が切れた (または危険にさらされた) 場合、発行者が主張して透明性サービスに登録される時刻よりも前に失効情報が公開されていないことを確認することで、ステートメントの信頼性を確認できます。

This new usage scenario motivates the "Timestamp, Then COSE" mode defined in Section 2.2.

この新しい使用シナリオは、セクション 2.2 で定義されている「タイムスタンプ、次に COSE」モードを動機付けます。

3. Timestamp Tokens per RFC 3161: COSE Header Parameters
3. RFC 3161 に準拠したタイムスタンプ トークン: COSE ヘッダー パラメーター

The two modes described in Sections 2.2 and 2.1 use different inputs into the timestamping machinery and consequently create different kinds of bindings between COSE and TST. To clearly separate their semantics, two different COSE header parameters are defined as described in the following subsections.

セクション 2.2 および 2.1 で説明されている 2 つのモードは、タイムスタンプ機構への異なる入力を使用するため、COSE と TST の間に異なる種類のバインディングが作成されます。セマンティクスを明確に区別するために、次のサブセクションで説明するように、2 つの異なる COSE ヘッダー パラメーターが定義されます。

3.1. 3161-ctt
3.1. 3161-ctt

The 3161-ctt COSE _unprotected_ header parameter MUST be used for the mode described in Section 2.1.

セクション 2.1 で説明されているモードには、3161-ctt COSE _unprotected_ ヘッダー パラメーターを使用しなければなりません (MUST)。

The 3161-ctt unprotected header parameter contains a DER-encoded TST [RFC3161] wrapped in a CBOR byte string (Major type 2).

3161-ctt の保護されていないヘッダー パラメーターには、CBOR バイト文字列 (メジャー タイプ 2) でラップされた DER でエンコードされた TST [RFC3161] が含まれています。

The MessageImprint sent in the request to the TSA MUST be

TSA へのリクエストで送信される MessageImprint は、

* the hash of the CBOR-encoded signature field of the COSE_Sign1 message, or

* COSE_Sign1 メッセージの CBOR エンコードされた署名フィールドのハッシュ、または

* the hash of the CBOR-encoded signatures field of the COSE_Sign message.

* COSE_Sign メッセージの CBOR でエンコードされた署名フィールドのハッシュ。

In either case, to minimize dependencies, the hash algorithm SHOULD be the same as the algorithm used for signing the COSE message. This may not be possible if the timestamp token has been obtained outside the processing context in which the COSE object is assembled.

いずれの場合も、依存関係を最小限に抑えるために、ハッシュ アルゴリズムは COSE メッセージの署名に使用されるアルゴリズムと同じである必要があります (SHOULD)。COSE オブジェクトがアセンブルされる処理コンテキストの外部でタイムスタンプ トークンが取得された場合、これは不可能な場合があります。

Refer to Sections 3.1.1 and 3.1.2 for concrete examples of MessageImprint computation.

MessageImprint 計算の具体的な例については、セクション 3.1.1 および 3.1.2 を参照してください。

4. Timestamp Processing
4. タイムスタンプの処理

Timestamp tokens [RFC3161] use Cryptographic Message Syntax (CMS) as the signature envelope format. [RFC5652] provides details about signature verification, and [RFC3161] provides details specific to timestamp token validation. The payload of the signed timestamp token is the TSTInfo structure defined in [RFC3161], which contains the MessageImprint that was sent to the TSA. The hash algorithm is contained in the MessageImprint structure, together with the hash itself.

タイムスタンプ トークン [RFC3161] は、署名エンベロープ形式として暗号メッセージ構文 (CMS) を使用します。[RFC5652] は署名検証に関する詳細を提供し、[RFC3161] はタイムスタンプ トークンの検証に固有の詳細を提供します。署名付きタイムスタンプ トークンのペイロードは、[RFC3161] で定義されている TSTInfo 構造体であり、TSA に送信された MessageImprint が含まれています。ハッシュ アルゴリズムは、ハッシュ自体とともに MessageImprint 構造に含まれています。

As part of the signature verification, the receiver MUST make sure that the MessageImprint in the embedded timestamp token matches a hash of either the payload, signature, or signature fields, depending on the mode of use and type of COSE structure.

署名検証の一環として、受信者は、埋め込まれたタイムスタンプ トークン内の MessageImprint が、使用モードと COSE 構造の種類に応じて、ペイロード、署名、または署名フィールドのいずれかのハッシュと一致することを確認しなければなりません (MUST)。

Appendix B of [RFC3161] provides an example that illustrates how timestamp tokens can be used to verify signatures of a timestamped message when utilizing X.509 certificates.

[RFC3161] の付録 B には、X.509 証明書を利用する際にタイムスタンプ トークンを使用してタイムスタンプ付きメッセージの署名を検証する方法を示す例が提供されています。

5. Security Considerations
5. セキュリティに関する考慮事項

Please review the Security Considerations section in [RFC3161]; these considerations apply to this document as well.

[RFC3161] のセキュリティに関する考慮事項のセクションを参照してください。これらの考慮事項はこのドキュメントにも当てはまります。

Also review the Security Considerations section in [RFC9052]. These considerations apply to this document as well, particularly with regard to the need for implementations to protect private key material. Additionally, solutions based on the COSE header parameters defined in this document must be able to report compromised keys promptly.

[RFC9052] のセキュリティに関する考慮事項のセクションも参照してください。これらの考慮事項は、特に秘密鍵マテリアルを保護するための実装の必要性に関して、この文書にも当てはまります。さらに、この文書で定義されている COSE ヘッダー パラメーターに基づくソリューションは、侵害されたキーを迅速に報告できなければなりません。

The following scenario assumes that an attacker can manipulate the clocks on the COSE signer and its relying parties, but not the TSA. It is also assumed that the TSA is a trusted third party, so the attacker cannot impersonate the TSA and create valid timestamp tokens. In such a setting, any tampering with the COSE signer's clock does not have an impact, because once the timestamp is obtained from the TSA, it becomes the only reliable source of time. However, in both CTT mode and TTC mode, a denial of service can occur if the attacker can adjust the relying party's clock so that the CMS validation fails. This could disrupt the timestamp validation.

次のシナリオは、攻撃者が COSE 署名者とその依存者のクロックを操作できるが、TSA は操作できないことを想定しています。また、TSA は信頼できる第三者であると想定されているため、攻撃者が TSA になりすまして有効なタイムスタンプ トークンを作成することはできません。このような設定では、TSA からタイムスタンプを取得すると、それが唯一の信頼できる時刻情報源となるため、COSE 署名者の時計が改ざんされても影響はありません。ただし、CTT モードと TTC モードの両方で、攻撃者が証明書利用者のクロックを調整して CMS 検証が失敗する可能性がある場合、サービス妨害が発生する可能性があります。これにより、タイムスタンプの検証が中断される可能性があります。

In CTT mode, an attacker could manipulate the unprotected header by removing or replacing the timestamp. To avoid that, the COSE Signed Message should be integrity protected during transit and at rest.

CTT モードでは、攻撃者はタイムスタンプを削除または置き換えることにより、保護されていないヘッダーを操作する可能性があります。これを回避するには、COSE 署名付きメッセージの転送中および保存中の整合性を保護する必要があります。

In TTC mode, the TSA is given an opaque identifier (a cryptographic hash value) for the payload. While this means that the content of the payload is not directly revealed, to prevent comparison with known payloads or disclosure of identical payloads being used over time, the payload would need to be armored, e.g., with a nonce that is shared with the recipient of the header parameter but not the TSA. Such a mechanism is out of scope for this document.

TTC モードでは、TSA にはペイロードの不透明な識別子 (暗号化ハッシュ値) が与えられます。これは、ペイロードの内容が直接公開されないことを意味しますが、既知のペイロードとの比較や、使用されている同一のペイロードの長期的な開示を防ぐために、ペイロードは、たとえば TSA ではなくヘッダー パラメータの受信者と共有されるノンスなどで保護される必要があります。このようなメカニズムは、このドキュメントの範囲外です。

The resolution, accuracy, and precision of the TSA clock, as well as the expected latency introduced by round trips to and from the TSA, must be taken into account when implementing solutions based on the COSE header parameters defined in this document.

このドキュメントで定義されている COSE ヘッダー パラメーターに基づいてソリューションを実装する場合は、TSA クロックの分解能、精度、正確さ、および TSA との往復によって生じる予想される遅延を考慮する必要があります。

5.1. Avoiding Semantic Confusion
5.1. 意味上の混乱を避ける

CTT mode and TTC mode have different semantic meanings. An implementation must ensure that the contents of the CTT and TTC headers are interpreted according to their specific semantics. In particular, symmetric to the signature and assembly mechanics, each mode has its own separate verification algorithm.

CTT モードと TTC モードは異なる意味を持ちます。実装では、CTT ヘッダーと TTC ヘッダーの内容がそれぞれの特定のセマンティクスに従って解釈されることを保証する必要があります。特に、署名とアセンブリの仕組みに対して対称的に、各モードには独自の個別の検証アルゴリズムがあります。

Implementers MUST clearly differentiate between TSA timestamps [RFC3161] proving the existence of payload data at an earlier point in time (TTC) and timestamps explicitly providing evidence of the existence of the cryptographic signature (CTT). Failure to clearly distinguish between these timestamp semantics can result in vulnerabilities, such as incorrectly accepting signatures created after key revocation based on older payload-only timestamps. Validators must not interpret protected-header payload timestamps as proof of signature creation time and should rely exclusively on TSA timestamps [RFC3161] explicitly covering signature data for determining signature validity timing.

実装者は、以前の時点 (TTC) でのペイロード データの存在を証明する TSA タイムスタンプ [RFC3161] と、暗号署名の存在の証拠を明示的に提供するタイムスタンプ (CTT) を明確に区別しなければなりません (MUST)。これらのタイムスタンプ セマンティクスを明確に区別できないと、古いペイロードのみのタイムスタンプに基づいてキーの失効後に作成された署名が誤って受け入れられるなどの脆弱性が発生する可能性があります。検証者は、保護されたヘッダーのペイロードのタイムスタンプを署名作成時刻の証明として解釈してはならず、署名の有効性のタイミングを決定するために、署名データを明示的にカバーする TSA タイムスタンプ [RFC3161] のみに依存する必要があります。
