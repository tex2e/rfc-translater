Use of ML-KEM in the Cryptographic Message Syntax (CMS)
暗号メッセージ構文 (CMS) での ML-KEM の使用
Abstract
概要

Module-Lattice-Based Key-Encapsulation Mechanism (ML-KEM) is a quantum-resistant Key Encapsulation Mechanism (KEM). Three parameter sets for the ML-KEM algorithm are specified by the US National Institute of Standards and Technology (NIST) in FIPS 203. In order of increasing security strength (and decreasing performance), these parameter sets are ML-KEM-512, ML-KEM-768, and ML-KEM-1024. This document specifies the conventions for using ML-KEM with the Cryptographic Message Syntax (CMS) using the KEMRecipientInfo structure defined in "Using Key Encapsulation Mechanism (KEM) Algorithms in the Cryptographic Message Syntax (CMS)" (RFC 9629).

モジュール格子ベースのキー カプセル化メカニズム (ML-KEM) は、耐量子性キー カプセル化メカニズム (KEM) です。ML-KEM アルゴリズムの 3 つのパラメーター セットは、米国国立標準技術研究所 (NIST) の FIPS 203 で指定されています。これらのパラメーター セットは、セキュリティ強度が高まる (およびパフォーマンスが低下する) 順に、ML-KEM-512、ML-KEM-768、および ML-KEM-1024 です。この文書では、「暗号メッセージ構文 (CMS) でのキー カプセル化メカニズム (KEM) アルゴリズムの使用」 (RFC 9629) で定義されている KEMRecipientInfo 構造を使用して、暗号メッセージ構文 (CMS) で ML-KEM を使用するための規則を指定します。

1. Introduction
1. はじめに

The Module-Lattice-Based Key-Encapsulation Mechanism (ML-KEM) is an IND-CCA2-secure Key Encapsulation Mechanism (KEM) standardized in [FIPS203] by the NIST PQC Project [NIST-PQ]. ML-KEM is the name given to the final standardized version and is incompatible with pre-standards versions, often called "Kyber".

モジュール格子ベースのキーカプセル化メカニズム (ML-KEM) は、NIST PQC プロジェクト [NIST-PQ] によって [FIPS203] で標準化された IND-CCA2 で安全なキーカプセル化メカニズム (KEM) です。ML-KEM は、最終的な標準化バージョンに付けられた名前であり、「Kyber」と呼ばれることが多い標準以前のバージョンとは互換性がありません。

[RFC9629] defines the KEMRecipientInfo structure for the use of KEM algorithms for the CMS enveloped-data content type, the CMS authenticated-data content type, and the CMS authenticated-enveloped-data content type. This document specifies the direct use of ML-KEM in the KEMRecipientInfo structure using each of the three parameter sets from [FIPS203], namely ML-KEM-512, ML-KEM-768, and ML-KEM-1024. It does not address or preclude the use of ML-KEM as part of any hybrid scheme.

[RFC9629] は、CMS エンベロープ データ コンテンツ タイプ、CMS 認証済みデータ コンテンツ タイプ、および CMS 認証済みエンベロープ データ コンテンツ タイプの KEM アルゴリズムを使用するための KEMRecipientInfo 構造を定義しています。この文書は、[FIPS203] の 3 つのパラメータ セット、つまり ML-KEM-512、ML-KEM-768、および ML-KEM-1024 のそれぞれを使用して、KEMRecipientInfo 構造で ML-KEM を直接使用することを指定します。ハイブリッド スキームの一部として ML-KEM の使用を取り上げたり、排除したりするものではありません。

1.2. ML-KEM
1.2. ML-KEM

ML-KEM is a lattice-based KEM using Module Learning with Errors as its underlying primitive, which is a structured lattices variant that offers good performance and relatively small and balanced key and ciphertext sizes. ML-KEM was standardized with three parameter sets: ML-KEM-512, ML-KEM-768, and ML-KEM-1024. The parameters for each of the security levels were chosen to be at least as secure as a generic block cipher of 128, 192, or 256 bits, respectively. Appendix B provides more information on ML-KEM security levels and sizes.

ML-KEM は、基礎となるプリミティブとしてエラーを伴うモジュール学習を使用する格子ベースの KEM です。これは、優れたパフォーマンスと、比較的小さくバランスのとれたキーと暗号文のサイズを提供する構造化格子のバリアントです。ML-KEM は、ML-KEM-512、ML-KEM-768、および ML-KEM-1024 の 3 つのパラメータ セットで標準化されました。各セキュリティ レベルのパラメータは、それぞれ 128、192、または 256 ビットの汎用ブロック暗号と少なくとも同程度の安全性が確保されるように選択されています。付録 B では、ML-KEM のセキュリティ レベルとサイズに関する詳細情報を提供します。

All KEM algorithms provide three functions: KeyGen(), Encapsulate(), and Decapsulate().

すべての KEM アルゴリズムは、KeyGen()、Encapsulate()、および Decapsulate() の 3 つの関数を提供します。

The following summarizes these three functions for the ML-KEM algorithm, referencing corresponding functions in [FIPS203]:

以下は、[FIPS203] の対応する関数を参照しながら、ML-KEM アルゴリズムのこれら 3 つの関数を要約しています。

KeyGen() -> (ek, dk):

KeyGen() -> (ek, dk):

Generate the public encapsulation key (ek) and a private decapsulation key (dk). [FIPS203] specifies two formats for an ML-KEM private key: a 64-octet seed (d,z) and an (expanded) private decapsulation key (dk). Algorithm 19 (ML-KEM.KeyGen()) from [FIPS203] generates the public encapsulation key (ek) and the private decapsulation key (dk). As an alternative, when a seed (d,z) is generated first and then the seed is expanded to get the keys, algorithm 16 (ML-KEM.KeyGen_internal(d,z)) from [FIPS203] expands the seed to ek and dk. See Section 6 of [RFC9935] for private key encoding considerations.

公開カプセル化キー (ek) と秘密のカプセル化解除キー (dk) を生成します。[FIPS203] では、ML-KEM 秘密鍵の 2 つの形式、64 オクテットのシード (d,z) と (拡張された) カプセル化解除秘密鍵 (dk) を指定しています。[FIPS203] のアルゴリズム 19 (ML-KEM.KeyGen()) は、公開カプセル化キー (ek) と秘密カプセル化解除キー (dk) を生成します。別の方法として、シード (d,z) が最初に生成され、その後シードが拡張されて鍵が取得される場合、[FIPS203] のアルゴリズム 16 (ML-KEM.KeyGen_internal(d,z)) はシードを ek と dk に拡張します。秘密鍵エンコードに関する考慮事項については、[RFC9935] のセクション 6 を参照してください。

Encapsulate(ek) -> (c, ss):

カプセル化(ek) -> (c, ss):

Given the recipient's public key (ek), produce both a ciphertext (c) to be passed to the recipient and a shared secret (ss) for use by the originator. Algorithm 20 (ML-KEM.Encaps(ek)) from [FIPS203] is the encapsulation function for ML-KEM.

受信者の公開鍵 (ek) を指定して、受信者に渡す暗号文 (c) と発信者が使用する共有秘密 (ss) の両方を生成します。[FIPS203] のアルゴリズム 20 (ML-KEM.Encaps(ek)) は、ML-KEM のカプセル化関数です。

Decapsulate(dk, c) -> ss:

カプセル化解除(dk, c) -> ss:

Given the private key (dk) and the ciphertext (c), produce the shared secret (ss) for the recipient. Algorithm 21 (ML-KEM.Decaps(dk,c)) from [FIPS203] is the decapsulation function for ML-KEM. If the private key is stored in seed form, ML-KEM.KeyGen_internal(d,z) may be needed as a first step to compute dk. See Section 8 of [RFC9935] for consistency considerations if the private key was stored in both seed and expanded formats.

秘密キー (dk) と暗号文 (c) を指定して、受信者の共有秘密 (ss) を生成します。[FIPS203] のアルゴリズム 21 (ML-KEM.Decaps(dk,c)) は、ML-KEM のカプセル化解除関数です。秘密キーがシード形式で保存されている場合、dk を計算するための最初のステップとして ML-KEM.KeyGen_internal(d,z) が必要になる場合があります。秘密鍵がシード形式と拡張形式の両方で保存された場合の一貫性に関する考慮事項については、[RFC9935] のセクション 8 を参照してください。

All security levels of ML-KEM use SHA3-256, SHA3-512, SHAKE128, and SHAKE256 internally.

ML-KEM のすべてのセキュリティ レベルは、内部的に SHA3-256、SHA3-512、SHAKE128、および SHAKE256 を使用します。

2. Use of the ML-KEM Algorithm in the CMS
2. CMS での ML-KEM アルゴリズムの使用

The ML-KEM algorithm MAY be employed for one or more recipients in the CMS enveloped-data content type [RFC5652], the CMS authenticated-data content type [RFC5652], or the CMS authenticated-enveloped-data content type [RFC5083]. In each case, the KEMRecipientInfo [RFC9629] is used with the ML-KEM algorithm to securely transfer the content-encryption key from the originator to the recipient.

ML-KEM アルゴリズムは、CMS エンベロープ データ コンテンツ タイプ [RFC5652]、CMS 認証データ コンテンツ タイプ [RFC5652]、または CMS 認証エンベロープ データ コンテンツ タイプ [RFC5083] の 1 人以上の受信者に使用できます (MAY)。いずれの場合も、KEMRecipientInfo [RFC9629] は ML-KEM アルゴリズムとともに使用され、コンテンツ暗号化キーを発信者から受信者に安全に転送します。

Processing ML-KEM with KEMRecipientInfo follows the same steps as Section 2 of [RFC9629]. To support the ML-KEM algorithm, a CMS originator MUST implement the Encapsulate() function and a CMS recipient MUST implement the Decapsulate() function.

KEMRecipientInfo を使用した ML-KEM の処理は、[RFC9629] のセクション 2 と同じ手順に従います。ML-KEM アルゴリズムをサポートするには、CMS 発信者は Encapsulate() 関数を実装しなければならず、CMS 受信者は Decapsulate() 関数を実装しなければなりません。

2.1. RecipientInfo Conventions
2.1. RecipientInfo の規約

When the ML-KEM algorithm is employed for a recipient, the RecipientInfo alternative for that recipient MUST be OtherRecipientInfo using the KEMRecipientInfo structure as defined in [RFC9629].

ML-KEM アルゴリズムが受信者に使用される場合、その受信者に対する RecipientInfo の代替は、[RFC9629] で定義されている KEMRecipientInfo 構造を使用する OtherRecipientInfo でなければなりません (MUST)。

The fields of the KEMRecipientInfo have the following meanings:

KEMRecipientInfo のフィールドには次の意味があります。

バージョン

The syntax version number; it MUST be 0.

構文のバージョン番号。0でなければなりません。

rid

取り除く

Identifies the recipient's certificate or public key.

受信者の証明書または公開キーを識別します。

kem

ケム

Identifies the KEM algorithm; it MUST contain one of id-alg-ml-kem-512, id-alg-ml-kem-768, or id-alg-ml-kem-1024. These identifiers are reproduced in Section 3.

KEM アルゴリズムを識別します。id-alg-ml-kem-512、id-alg-ml-kem-768、または id-alg-ml-kem-1024 のいずれかを含める必要があります。これらの識別子はセクション 3 で再現されます。

kemct

ケムクト

The ciphertext produced for this recipient.

この受信者用に生成された暗号文。

kdf

kdf

Identifies the key derivation algorithm. Note that the Key Derivation Function (KDF) used for CMS RecipientInfo process MAY be different than the KDF used within the ML-KEM algorithm. Implementations MUST support the HMAC-based Key Derivation Function (HKDF) [RFC5869] with SHA-256 [FIPS180] using the id-alg-hkdf-with-sha256 KDF object identifier (OID) [RFC8619]. As specified in [RFC8619], the parameter field MUST be absent when this OID appears within the ASN.1 type AlgorithmIdentifier. Implementations MAY support other KDFs as well.

キー導出アルゴリズムを識別します。CMS RecipientInfo プロセスに使用されるキー導出関数 (KDF) は、ML-KEM アルゴリズム内で使用される KDF と異なる場合があることに注意してください。実装は、id-alg-hkdf-with-sha256 KDF オブジェクト識別子 (OID) [RFC8619] を使用して、SHA-256 [FIPS180] を備えた HMAC ベースの鍵導出関数 (HKDF) [RFC5869] をサポートしなければなりません (MUST)。[RFC8619] で規定されているように、この OID が ASN.1 タイプの AlgorithmIdentifier 内に出現する場合、パラメータフィールドは存在しなければなりません (MUST)。実装では他の KDF もサポートしてもよい (MAY)。

キー暗号化キーのサイズ (オクテット単位)。

うっかり

KDF へのオプションの入力。CMS での ML-KEM の安全な使用は、ukm 値の使用に依存しないため、このドキュメントではこの値にいかなる要件も課しません。ukm パラメータの詳細については、[RFC9629] のセクション 3 を参照してください。

2.2.1. Use of the HKDF-Based Key Derivation Function
2.2.1. HKDF ベースのキー導出関数の使用

The HKDF function is a composition of the HKDF-Extract and HKDF-Expand functions.

HKDF 関数は、HKDF-Extract 関数と HKDF-Expand 関数を組み合わせたものです。 

   HKDF(salt, IKM, info, L)
     = HKDF-Expand(HKDF-Extract(salt, IKM), info, L)

When used with KEMRecipientInfo, the salt parameter is unused; that is, it is the zero-length string "". The IKM, info, and L parameters correspond to the same KDF inputs from Section 5 of [RFC9629]. The info parameter is independently generated by the originator and recipient. Implementations MUST confirm that L is consistent with the key size of the key-encryption algorithm.

KEMRecipientInfo と一緒に使用する場合、salt パラメータは使用されません。つまり、長さ 0 の文字列 "" です。IKM、info、および L パラメータは、[RFC9629] のセクション 5 からの同じ KDF 入力に対応します。info パラメータは、発信者と受信者によって個別に生成されます。実装では、L が鍵暗号化アルゴリズムの鍵サイズと一致していることを確認しなければなりません (MUST)。

2.3. Certificate Conventions
2.3. 証明書の規約

[RFC5280] specifies the profile for using X.509 certificates in Internet applications. A recipient static public key is needed for ML-KEM and the originator obtains that public key from the recipient's certificate. The conventions for carrying ML-KEM public keys are specified in [RFC9935].

[RFC5280] は、インターネット アプリケーションで X.509 証明書を使用するためのプロファイルを指定します。ML-KEM には受信者の静的公開キーが必要であり、発信者は受信者の証明書からその公開キーを取得します。ML-KEM 公開鍵を運ぶための規則は [RFC9935] で指定されています。

2.4. SMIME Capabilities Attribute Conventions
2.4. SMIME 機能の属性規則

Section 2.5.2 of [RFC8551] defines the SMIMECapabilities attribute to announce a partial list of algorithms that an S/MIME implementation can support. 4. Security Considerations
4. セキュリティに関する考慮事項

The Security Considerations sections of [RFC9935] and [RFC9629] apply to this specification as well.

[RFC9935] および [RFC9629] のセキュリティに関する考慮事項のセクションは、この仕様にも同様に適用されます。

For ongoing discussions of ML-KEM-specific security considerations, refer to [MLKEM-SEC-CONS].

ML-KEM 固有のセキュリティに関する考慮事項についての継続的な議論については、[MLKEM-SEC-CONS] を参照してください。

Implementations MUST protect the ML-KEM private key, the key-encryption key, the content-encryption key, message-authentication key, and the content-authenticated-encryption key. Of these keys, all but the private key are ephemeral and MUST be wiped after use. Disclosure of the ML-KEM private key could result in the compromise of all messages protected with that key. Disclosure of the key-encryption key, the content-encryption key, or the content-authenticated-encryption key could result in the compromise of the associated encrypted content. Disclosure of the key-encryption key, the message-authentication key, or the content-authenticated-encryption key could allow modification of the associated authenticated content.

実装では、ML-KEM 秘密キー、キー暗号化キー、コンテンツ暗号化キー、メッセージ認証キー、およびコンテンツ認証暗号化キーを保護しなければなりません (MUST)。これらの鍵のうち、秘密鍵を除くすべては一時的なものであり、使用後は消去する必要があります。ML-KEM 秘密キーが公開されると、そのキーで保護されているすべてのメッセージが侵害される可能性があります。キー暗号化キー、コンテンツ暗号化キー、またはコンテンツ認証暗号化キーが開示されると、関連する暗号化コンテンツが侵害される可能性があります。キー暗号化キー、メッセージ認証キー、またはコンテンツ認証暗号化キーの開示により、関連する認証済みコンテンツの変更が可能になる可能性があります。

Additional considerations related to key management may be found in [NIST.SP.800-57pt1r5].

鍵管理に関する追加の考慮事項は、[NIST.SP.800-57pt1r5] に記載されています。

The generation of private keys relies on random numbers, as does the encapsulation function of ML-KEM. The use of inadequate pseudorandom number generators (PRNGs) to generate these values can result in little or no security. In the case of key generation, a random 32-byte seed is used to deterministically derive the key (with an additional 32 bytes reserved as a rejection value). In the case of encapsulation, a KEM is derived from the underlying ML-KEM public key encryption algorithm by deterministically encrypting a random 32-byte message for the public key. If the random value is weakly chosen, then an attacker may find it much easier to reproduce the PRNG environment that produced the keys or ciphertext, searching the resulting small set of possibilities for a matching public key or ciphertext value, rather than performing a more complex algorithmic attack against ML-KEM. The generation of quality random numbers is difficult; see Section 3.3 of [FIPS203] for some additional information.

秘密キーの生成は、ML-KEM のカプセル化機能と同様に、乱数に依存します。これらの値を生成するために不適切な擬似乱数ジェネレーター (PRNG) を使用すると、セキュリティがほとんど、またはまったく得られない可能性があります。キー生成の場合、ランダムな 32 バイトのシードを使用してキーを決定的に導出します (追加の 32 バイトは拒否値として予約されています)。カプセル化の場合、KEM は、公開キーのランダムな 32 バイト メッセージを決定論的に暗号化することによって、基礎となる ML-KEM 公開キー暗号化アルゴリズムから派生します。ランダム値が弱く選択されている場合、攻撃者は、ML-KEM に対してより複雑なアルゴリズム攻撃を実行するよりも、キーまたは暗号文を生成した PRNG 環境を再現し、結果として得られる少数の可能性のセットを検索して、一致する公開キーまたは暗号文値を探す方がはるかに簡単であると判断する可能性があります。高品質の乱数を生成するのは困難です。追加情報については、[FIPS203] のセクション 3.3 を参照してください。

ML-KEM encapsulation and decapsulation only outputs a shared secret and ciphertext. Implementations MUST NOT use intermediate values directly for any purpose.

ML-KEM のカプセル化とカプセル化解除は、共有秘密と暗号文のみを出力します。実装では、いかなる目的でも中間値を直接使用してはなりません。

Implementations SHOULD NOT reveal information about intermediate values or calculations, whether by timing or other "side channels"; otherwise, an opponent may be able to determine information about the keying data and/or the recipient's private key. Although not all intermediate information may be useful to an opponent, it is preferable to conceal as much information as is practical, unless analysis specifically indicates that the information would not be useful to an opponent.

実装は、タイミングや他の「サイドチャネル」によっても、中間値や計算に関する情報を明らかにしてはなりません。そうしないと、対戦相手がキーイング データおよび/または受信者の秘密鍵に関する情報を特定できる可能性があります。すべての中間情報が対戦相手にとって有益であるとは限りませんが、分析によってその情報が対戦相手にとって有益ではないことが特に示されない限り、実用的な情報はできるだけ隠すことが望ましいです。

Generally, good cryptographic practice employs a given ML-KEM key pair in only one scheme. This practice avoids the risk that vulnerability in one scheme may compromise the security of the other and may be essential to maintain provable security.

一般に、適切な暗号化手法では、特定の ML-KEM キー ペアを 1 つのスキームのみで使用します。この実践は、一方のスキームの脆弱性がもう一方のスキームのセキュリティを侵害する可能性があるというリスクを回避し、証明可能なセキュリティを維持するために不可欠である可能性があります。

Parties can gain assurance that implementations are correct through formal implementation validation, such as the NIST Cryptographic Module Validation Program (CMVP) [CMVP].

当事者は、NIST 暗号モジュール検証プログラム (CMVP) [CMVP] などの正式な実装検証を通じて、実装が正しいという保証を得ることができます。
